## Учебный проект по React для Senior Java разработчика

Этот проект — пошаговое учебное задание, ориентированное на разработчика с сильным опытом в Java и базовыми знаниями JavaScript. Цель — **освоить современный React** (включая все основные хуки и типичную архитектуру крупного фронтенд-проекта), а не просто «запустить пример».

Ниже:
- **Часть 1** — быстрый старт, все нужные команды и файлы (Vite, npm, TypeScript).
- **Часть 2** — концепция учебного приложения.
- **Часть 3** — архитектура проекта, слои и структура папок.
- **Часть 4** — пошаговый план задач с указанием, какой хук/механизм использовать.
- **Часть 5** — описание внешних API (ендпойнты, методы, структура ответа).

В этом файле **нет готовых React-компонентов и JSX-кода** — нужно будет подумать и реализовать самому. Здесь только подсказки, структура и требования.

## Часть 1. Быстрый старт: npm, Vite, TypeScript, tsc

### 1. Что такое npm

- **npm** — пакетный менеджер для экосистемы Node.js.
- Он:
  - устанавливает зависимости проекта (React, Vite и т.п.),
  - управляет скриптами сборки/запуска (через файл `package.json`),
  - позволяет публиковать и использовать пакеты.

У тебя уже есть **npm 10.9.4**, этого достаточно.

### 2. Что такое Vite и зачем он нужен

- **Vite** — это современный инструмент для разработки SPA:
  - поднимает **dev-сервер** с горячей перезагрузкой (HMR),
  - собирает проект для продакшена (bundle/optimize),
  - по умолчанию хорошо интегрируется с **React** и **TypeScript**.
- Можно писать React и без Vite (например, с Webpack или вообще без бандлера), но:
  - в реальных коммерческих проектах **почти всегда** используется сборщик,
  - Vite — одно из самых популярных решений сейчас.

**Вывод:** для учебного и реального проекта React **Vite — разумный дефолтный выбор**.

### 3. Что такое TypeScript и tsc

- **TypeScript (TS)** — надстройка над JavaScript, добавляющая **статическую типизацию**.
- **`tsc`** — TypeScript Compiler, который:
  - проверяет типы,
  - компилирует `.ts`/`.tsx` в обычный JavaScript.
- В современном фронтенде:
  - использование TypeScript в React-проектах уже **де-факто стандарт**,
  - Vite умеет работать с TypeScript «из коробки».

**Вывод:** для учебного проекта имеет смысл **сразу использовать TypeScript**, чтобы привычный опыт Java с типами помогал, а не мешал.

### 4. Быстрый старт: создание проекта (с Vite + React + TypeScript)

#### 4.1. Инициализация проекта через Vite

Из директории `react-learning` (корень репозитория) выполни:


Далее:
1. Vite спросит **имя проекта** — можно указать `react-learning` или любое другое.
2. Выбери **шаблон**:
   - `React` → `TypeScript`.

Vite создаст папку проекта (если имя совпадает с текущей — файлы появятся в текущей папке).

> Если Vite предлагает запустить установку зависимостей автоматически — можно согласиться, но чтобы лучше понять, что происходит, полезно выполнить установку явно.

#### 4.2. Установка зависимостей

Перейди в папку проекта (если она вдруг другая, чем `react-learning` — используй своё имя):

```bash
cd react-learning
npm install
```

Это:
- создаст файл `package-lock.json`,
- скачает в папку `node_modules` все зависимости (React, ReactDOM, Vite, TypeScript и пр.).

#### 4.3. Основные файлы, которые должны появиться

После создания шаблона Vite + React + TS в корне проекта будут примерно такие важные файлы:

- `package.json` — описание проекта, список зависимостей и npm-скриптов.
- `tsconfig.json` — конфигурация TypeScript.
- `vite.config.ts` — конфигурация Vite.
- `index.html` — HTML-шаблон, в который монтируется React-приложение.
- `src/main.tsx` — точка входа в приложение (рендер корневого React-компонента).
- `src/App.tsx` — корневой компонент приложения.

Ниже — **структура и назначение** этих файлов (без JSX-кода и без подробной реализации компонентов).

#### 4.4. Структура и пояснения к `package.json`

Файл `package.json`:
- содержит:
  - имя и версию проекта,
  - список зависимостей,
  - список **npm-скриптов** в секции `scripts`.
- именно скрипты ты будешь запускать командами `npm run <имя_скрипта>`.

Типичные скрипты, которые должен содержать твой `package.json`:
- **`dev`** — запускает dev-сервер Vite.
- **`build`** — собирает приложение для продакшена.
- **`preview`** — запускает локальный сервер для просмотра собранной версии.
- **`lint`** (опционально) — запускает линтер.

Твоя задача:
- открыть `package.json`,
- внимательно прочитать, какие скрипты уже есть,
- понять, какой скрипт за что отвечает.

**Что важно усвоить:**
- Команда `npm run dev` → вызывает скрипт `dev` → запускает Vite → поднимает сервер (по умолчанию `http://localhost:5173`).
- Команда `npm run build` → вызывает Vite в режиме сборки → результат собирается в папку `dist`.

#### 4.5. Файл `tsconfig.json`

`tsconfig.json`:
- определяет:
  - целевую версию JavaScript (`target`),
  - какие файлы включать/исключать,
  - строгий режим типов и т.д.

Задача:
- открыть `tsconfig.json`,
- найти ключи вроде `strict`, `jsx`, `module`, `target`,
- понять, как они соотносятся с привычными понятиями из Java (уровень строгости, версия байткода и т.п.).

#### 4.6. Файл `vite.config.ts`

В этом файле:
- подключаются плагины (например, плагин для React),
- настраиваются alias-ы для импортов, порты и т.п.

Задача:
- понять, как Vite «узнает», что проект — React-проект с TypeScript.

#### 4.7. Файл `index.html`

Это обычный HTML-файл, в котором есть:
- `<div id="root"></div>` (или похожий),
- подключение скрипта, который будет загружать React-часть.

Важный момент:
- В отличие от классического HTML+JS, ты **не наполняешь этот HTML разметкой руками**.
- Вместо этого React монтируется в контейнер (`<div id="root">`) и управляет DOM сам.

#### 4.8. Файл `src/main.tsx`

Точка входа:
- импортирует корневой компонент (обычно `App`),
- вызывает ReactDOM для рендера в `#root`.

Твоя задача:
- открыть `main.tsx`,
- понять, как выглядит процесс монтирования,
- увидеть, где можно будет в будущем подключать глобальные провайдеры (например, контекст).

#### 4.9. Запуск проекта

После установки зависимостей:

```bash
npm run dev
```

- В консоли появится URL (обычно `http://localhost:5173`).
- Открой его в браузере — увидишь дефолтный шаблон React от Vite.

С этого момента можно:
- удалить/переписать шаблонные компоненты,
- начать строить собственную архитектуру (см. ниже).

## Часть 2. Концепция учебного приложения

Чтобы задействовать все основные механизмы React и сделать архитектуру, похожую на реальный проект, создадим **мини-систему управления задачами/проектами** (Task/Project Management Dashboard) с внешними данными.

### 1. Основная идея

Приложение — **панель управления задачами**:
- список пользователей (исполнителей),
- список задач,
- фильтрация/поиск,
- детали задачи,
- изменение статуса (например, «новая», «в работе», «завершена»),
- базовая авторизация-заглушка (имитация входа пользователя).

Цель:
- работать с состоянием (локальным и глобальным),
- обрабатывать эффекты (запросы к API),
- использовать контекст,
- работать с мемоизацией и колбэками,
- реализовать собственные хуки там, где это логично.

## Часть 3. Архитектура и структура проекта

### 1. Общая идея архитектуры

Сделаем архитектуру, похожую на типичные большие SPA:

- **`src/app`** — точка входа приложения и глобальные настройки.
- **`src/shared`** — общие (переиспользуемые) сущности:
  - UI-компоненты без бизнес-логики (кнопки, инпуты, модальные окна),
  - общие утилиты и хелперы,
  - константы, типы.
- **`src/entities`** — доменные сущности:
  - `user`, `task`, возможно `project`.
  - Каждая сущность может содержать:
    - типы данных,
    - функции работы с API (fetch/transform),
    - простые доменные компоненты (например, карточка задачи).
- **`src/features`** — отдельные фичи/возможности:
  - фильтр задач,
  - редактирование статуса задачи,
  - авторизация пользователя.
- **`src/widgets`** — более крупные компоновки:
  - панель фильтров,
  - таблица задач,
  - панель пользователя.
- **`src/pages`** — страницы:
  - главная страница дашборда,
  - страница деталей задачи,
  - страница «Вход» (логин).

Это схоже с подходами типа Feature-Sliced Design, но без строгого следования какому-то одному стандарту.

### 2. Пример структуры папок

Пример (без файлов компонента, которые ты напишешь сам):

- `src/`
  - `app/`
    - `App.tsx`
    - `providers/` (глобальные провайдеры, например, контекст авторизации)
    - `router/` (если захочешь добавить роутер на более позднем этапе)
  - `shared/`
    - `ui/` (кнопки, инпуты, селекты, спиннеры)
    - `lib/` (утилиты, функции работы с датами, хелперы)
    - `config/` (конфигурация, базовый URL API и т.п.)
  - `entities/`
    - `user/`
    - `task/`
  - `features/`
    - `task-filter/`
    - `task-edit-status/`
    - `auth/`
  - `widgets/`
    - `task-list/`
    - `task-details-panel/`
    - `user-panel/`
  - `pages/`
    - `DashboardPage/`
    - `TaskDetailsPage/`
    - `LoginPage/`

Твоя задача:
- создать эту структуру папок,
- разбить будущие компоненты по слоям, а не складывать всё в один `App.tsx`.

## Часть 4. Пошаговый учебный план и где использовать какие хуки

### 1. Базовый рендер и `useState`

**Цель:** понять базовый цикл рендера, работу с локальным состоянием.

Задания:
- Создать в `App.tsx` простой каркас:
  - глобальный layout (хедер/основная область).
- В одном из виджетов (например, `user-panel`) реализовать:
  - переключение «выбранного» пользователя с помощью **`useState`**.

Подсказки по `useState`:
- Используй там, где:
  - состояние **принадлежит одному компоненту**,
  - не нужно шарить его на много уровней глубже.
- Например:
  - выбранная вкладка,
  - локальное состояние формы (до отправки).

### 2. Загружать данные с API через `useEffect`

**Цель:** научиться работать с побочными эффектами и асинхронными запросами.

Задания:
- В сущности `user` и/или `task` создать модуль, который:
  - знает, **какие эндпойнты** использовать (см. Часть 5),
  - реализует функции загрузки списка пользователей/задач.
- В виджете/странице (например, `DashboardPage`) использовать:
  - **`useEffect`** для вызова функций загрузки при монтировании компонента,
  - локальные стейты для хранения:
    - списка задач,
    - статуса загрузки (loading/error/success),
    - выбранных фильтров (позже).

Подсказки по `useEffect`:
- Используй его для:
  - запросов к серверу при монтировании,
  - подписок на внешние ресурсы (WebSocket, события),
  - синхронизации с внешним миром (document title, localStorage).
- Важно:
  - внимательно продумать **зависимости** эффекта (массив зависимостей),
  - не вызывать эффекты без необходимости при каждом рендере.

### 3. Глобальное состояние авторизации через `useContext`

**Цель:** научиться делиться состоянием между многими компонентами без «проброса пропсов».

Задания:
- Реализовать «псевдо-авторизацию»:
  - в `features/auth` создать логику, которая:
    - хранит «текущего пользователя»,
    - умеет «логиниться» (просто выбирая пользователя из списка загруженных).
- В `app/providers` создать **контекст авторизации**:
  - тип значения контекста (например, текущий пользователь, флаг `isAuthenticated`, функции `login`/`logout`),
  - провайдер, который использует **`useState`** или **`useReducer`** внутри.
- Обернуть `App` в этот провайдер (скорее всего, в `main.tsx`).
- В `widgets/user-panel` или других компонентах:
  - получать данные о текущем пользователе из **контекста** (через `useContext`),
  - показывать/скрывать определённые элементы в зависимости от авторизации.

Подсказки по `useContext`:
- Используй, когда:
  - состояние/функции **логически глобальны** и нужны на многих уровнях дерева,
  - проброс пропсов становится громоздким.
- Не используйте контекст для всего подряд — это аналог «глобального стейта», имей в виду сложность отладки.

### 4. Работа с DOM напрямую через `useRef`

**Цель:** понять, как работать с DOM-элементами и сохранять произвольные значения между рендерами.

Задания:
- В `features/task-filter` использовать `useRef` для:
  - доступа к полю ввода фильтра (например, быстро очистить поле или поставить фокус),
  - измерения ширины/высоты какого-то блока (по желанию).
- В одной из фич (например, отслеживание количества рендеров или предыдущего значения) использовать `useRef` как:
  - контейнер для **мутабельного значения**, которое живёт между рендерами, но **не вызывает перерисовку**.

Подсказки по `useRef`:
- Подходит для:
  - хранения ссылки на DOM-элемент,
  - хранения «предыдущего значения» или любого объекта, который не должен триггерить ререндер.
- Не стоит использовать `useRef` как замену состоянию, когда нужно, чтобы UI реагировал на изменения.

### 5. Оптимизация рендеров с `useMemo` и `useCallback`

**Цель:** понять мемоизацию вычислений и колбэков.

Задания:
- В виджете `task-list`:
  - реализовать фильтрацию и сортировку задач,
  - использовать **`useMemo`** для вычисления:
    - отфильтрованного/отсортированного списка задач,
    - возможно, производных метрик (количество задач в каждом статусе).
- В компонентах, которые передают колбэки вниз по дереву (например, кнопки изменения статуса задачи):
  - обернуть обработчики в **`useCallback`**, чтобы:
    - избежать пересоздания функций на каждом рендере,
    - позволить дочерним компонентам использовать `React.memo` (если решишь его добавить).

Подсказки по `useMemo`:
- Используй, когда:
  - есть «дорогие» вычисления (фильтрации, агрегаты),
  - эти вычисления зависят от относительно небольшого набора данных/параметров.

Подсказки по `useCallback`:
- Используй, когда:
  - передаёшь функции как пропсы в глубину дерева,
  - хочешь избегать лишних рендеров дочерних компонентов из-за новых ссылок на функции.

Важно не переусердствовать:
- не нужно оборачивать в `useMemo`/`useCallback` всё подряд, только то, что **реально** создаёт проблемы производительности или читабельности.

### 6. Сложное состояние с `useReducer`

**Цель:** научиться управлять более сложными состояниями, похожими на редьюсеры в Redux.

Задания:
- Выбрать одну область состояния, которая:
  - достаточно сложна, чтобы `useState` стал громоздким (например, состояние фильтров задач + пагинация + сортировка),
  - или состояние авторизации/сессии.
- Реализовать **редьюсер**:
  - с отдельными action-ами (например, `SET_FILTER`, `RESET_FILTERS`, `SET_PAGE`, `SET_SORT`),
  - с типобезопасностью (TypeScript).
- Использовать **`useReducer`**:
  - в `features/task-filter`,
  - или в провайдере контекста авторизации.

Подсказки по `useReducer`:
- Подходит, когда:
  - состояния много и оно связано (несколько полей, зависящих друг от друга),
  - логика обновления сложнее, чем «просто присвоить новое значение».
- Похоже на:
  - классический `Reducer` из Java Stream API по духу (есть текущее состояние, есть action, есть новая версия состояния).

### 7. Кастомные хуки

**Цель:** научиться выносить переиспользуемую логику.

Задания:
- Выделить несколько сценариев, где логика:
  - повторяется в разных компонентах (например, загрузка и кеширование задач, работа с localStorage, общая форма).
- Создать кастомные хуки в `shared/lib/hooks` или в соответствующих слоях:
  - например, `useTasks`, `useUsers`, `useAuth`, `useDebouncedValue`.

Подсказки:
- Кастомный хук — это обычная функция, которая:
  - может использовать другие хуки,
  - инкапсулирует внутри сложную логику, возвращая наружу удобный интерфейс.

## Часть 5. Публичные API для использования в проекте

Ниже несколько доступных в интернете API, которые можно использовать для реализации задач. Все они не требуют авторизации по ключу (на момент написания), но ограничения/детали могут меняться, поэтому при необходимости стоит свериться с документацией.

### 1. JSONPlaceholder — задачи и пользователи

**Базовый URL:** `https://jsonplaceholder.typicode.com`

#### 1.1. Список пользователей

- **Endpoint:** `/users`
- **Метод:** `GET`
- **Заголовки:** обычно достаточно дефолтных (браузер сам отправит `Accept: application/json`).
- **Пример структуры ответа (упрощённо):**
  - массив объектов пользователя:
    - `id` — число,
    - `name` — полное имя,
    - `username` — логин,
    - `email` — email,
    - `address` — вложенный объект,
    - `company` — вложенный объект.

**Как использовать:**
- подгружать список пользователей при инициализации приложения,
- показывать их в `user-panel`,
- использовать `id` пользователя для фильтрации задач.

#### 1.2. Список задач (todos)

- **Endpoint:** `/todos`
- **Метод:** `GET`
- **Заголовки:** стандартные.
- **Структура ответа (упрощённо):**
  - массив объектов задачи:
    - `userId` — ID пользователя (связь с `/users`),
    - `id` — ID задачи,
    - `title` — заголовок,
    - `completed` — булево (выполнена/нет).

**Как использовать:**
- загружать список задач,
- связывать их с пользователями по `userId`,
- реализовать фильтрацию/отображение задач, изменение статуса (локально, т.к. API фиктивный).

### 2. ReqRes — имитация логина

**Базовый URL:** `https://reqres.in`

#### 2.1. Логин (успешный)

- **Endpoint:** `/api/login`
- **Метод:** `POST`
- **Тело запроса (JSON):**
  - `email` — строка,
  - `password` — строка.
- **Необходимые заголовки:**
  - `Content-Type: application/json`.
- **Структура ответа при успехе (упрощённо):**
  - объект с полем `token` — строка.

**Как использовать:**
- реализовать форму логина,
- при отправке формы:
  - отправить POST-запрос,
  - при успехе сохранить `token` (например, в контекст авторизации),
  - считать пользователя «авторизованным».

#### 2.2. Логин (ошибка)

- При некорректных данных вернётся ошибка (4xx),
- В теле может быть сообщение об ошибке.

**Как использовать:**
- показать обработку ошибок в UI,
- использовать `useEffect`/`useState` для управления состоянием загрузки/ошибки.

## Что делать дальше

1. **Создать проект через Vite** (React + TypeScript) и запустить `npm run dev`.
2. **Перестроить структуру `src`** под описанную архитектуру (app/shared/entities/features/widgets/pages).
3. **Реализовать доменные сущности** (`user`, `task`) и функции работы с API (JSONPlaceholder).

   - `useState` — локальные состояния,
   - `useEffect` — загрузка данных и побочные эффекты,
   - `useContext` — глобальное состояние авторизации,
   - `useRef` — работа с DOM и мутабельные значения,
   - `useMemo` / `useCallback` — оптимизация,
   - `useReducer` — сложное состояние,

   - убедиться, что бизнес-логика не размазана по случайным компонентам,

Главная цель — не просто «завести страницу», а **прочувствовать на практике ключевые механизмы React в контексте архитектуры, близкой к реальным большим фронтенд-проектам**.
