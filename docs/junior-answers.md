## JavaScript (основы)

### Типы данных и переменные

### 1. Какие типы данных существуют в JavaScript?

В JavaScript существует **8 типов данных**. Семь из них — примитивные: `string` (строка), `number` (число), `bigint` (большое целое число), `boolean` (логический тип: `true`/`false`), `undefined` (значение не присвоено), `null` (намеренное отсутствие значения) и `symbol` (уникальный идентификатор). Восьмой тип — `object` (объект), который является ссылочным (непримитивным) типом. Массивы, функции, даты и другие сложные структуры — всё это объекты «под капотом».

```js
typeof "hello"     // "string"
typeof 42          // "number"
typeof true        // "boolean"
typeof undefined   // "undefined"
typeof null        // "object"  ← исторический баг JS
typeof Symbol()    // "symbol"
typeof {}          // "object"
typeof []          // "object"
```

### 2. В чём разница между `var`, `let` и `const`?

`var` — устаревший способ объявления переменных. Он имеет **функциональную** область видимости и поднимается (hoisting) со значением `undefined`. `let` и `const` появились в ES6 и имеют **блочную** область видимости (внутри `{}`). `let` позволяет менять значение переменной, а `const` — нет (но если `const` ссылается на объект или массив, его содержимое можно изменять). Рекомендуется использовать `const` по умолчанию и `let`, когда нужно переприсвоение.

```js
var a = 1;   // функциональная область видимости, hoisting
let b = 2;   // блочная область видимости, можно менять
const c = 3; // блочная область видимости, нельзя переприсвоить

const arr = [1, 2];
arr.push(3); // допустимо — мы не меняем ссылку, а меняем содержимое
```

### 3. Что такое hoisting (поднятие)?

Hoisting — это механизм JavaScript, при котором объявления переменных и функций «поднимаются» в начало своей области видимости на этапе компиляции, ещё до выполнения кода. Переменные, объявленные через `var`, поднимаются со значением `undefined`. Функции, объявленные через `function declaration`, поднимаются целиком и могут вызываться до строки объявления. Переменные `let` и `const` тоже поднимаются, но находятся в «Temporal Dead Zone» и недоступны до строки объявления.

```js
console.log(x); // undefined (var поднимается)
var x = 5;

sayHi(); // "Привет!" (function declaration поднимается целиком)
function sayHi() {
  console.log("Привет!");
}

console.log(y); // ReferenceError (let в TDZ)
let y = 10;
```

### 4. В чём разница между `==` и `===`?

Оператор `==` (нестрогое сравнение) сравнивает значения **с приведением типов**. Это значит, что JavaScript автоматически пытается привести операнды к одному типу перед сравнением. Оператор `===` (строгое сравнение) сравнивает и значение, **и тип** без приведения. Рекомендуется всегда использовать `===`, чтобы избежать неожиданного поведения.

```js
0 == ""        // true  (оба приводятся к 0)
0 === ""       // false (число ≠ строка)
null == undefined  // true
null === undefined // false
1 == "1"       // true
1 === "1"      // false
```

### 5. Что такое `undefined` и `null`? В чём разница?

`undefined` означает, что переменная была объявлена, но ей **не было присвоено значение**. JavaScript автоматически присваивает `undefined` таким переменным. `null` — это **явное** указание на отсутствие значения, которое разработчик устанавливает намеренно. Оба они «falsy»-значения, но `typeof undefined` возвращает `"undefined"`, а `typeof null` — `"object"` (это известный исторический баг).

```js
let a;
console.log(a); // undefined — значение не присвоено

let b = null;
console.log(b); // null — намеренно пустое значение

console.log(null == undefined);  // true
console.log(null === undefined); // false
```

### 6. Что такое Temporal Dead Zone (TDZ)?

Temporal Dead Zone (временная мёртвая зона) — это промежуток от начала блока до строки, где переменная `let` или `const` фактически объявляется. Хотя переменная технически «поднята» (hoisted), обращение к ней в этой зоне вызывает `ReferenceError`. TDZ существует, чтобы помочь разработчикам находить ошибки — использование переменных до их объявления обычно указывает на логическую ошибку в коде.

```js
{
  // TDZ для переменной x начинается здесь
  console.log(x); // ReferenceError: Cannot access 'x' before initialization
  let x = 10;     // TDZ для x заканчивается здесь
}
```

### Функции

### 7. Чем отличается function declaration от function expression?

**Function declaration** — это объявление функции с помощью ключевого слова `function` в начале инструкции. Оно полностью поднимается (hoisted) и может вызываться до строки объявления. **Function expression** — это когда функция создаётся как часть выражения (например, присваивается переменной). Она не поднимается и доступна только после строки определения.

```js
// Function Declaration — можно вызвать до объявления
greet(); // "Привет!"
function greet() {
  console.log("Привет!");
}

// Function Expression — нельзя вызвать до объявления
sayBye(); // TypeError: sayBye is not a function
var sayBye = function () {
  console.log("Пока!");
};
```

### 8. Что такое стрелочные функции? Чем они отличаются от обычных?

Стрелочные функции (arrow functions) — это сокращённый синтаксис для записи функций, появившийся в ES6. Главное отличие: стрелочные функции **не имеют собственного `this`** — они берут `this` из окружающего контекста (лексический `this`). Также у них нет объекта `arguments`, они не могут быть использованы как конструкторы (с `new`) и не имеют `prototype`.

```js
const add = (a, b) => a + b;            // краткая запись с неявным return
const square = (x) => { return x * x; }; // с телом функции

const obj = {
  name: "Андрей",
  greet: function () {
    setTimeout(() => {
      console.log(this.name); // "Андрей" — this из окружающего контекста
    }, 100);
  },
};
```

### 9. Что такое замыкание (closure)?

Замыкание — это способность функции «запоминать» и обращаться к переменным из внешней (родительской) области видимости, даже после того как внешняя функция завершила выполнение. Каждый раз, когда создаётся функция, создаётся и замыкание. Это один из фундаментальных механизмов JavaScript, широко используемый для создания приватных переменных, фабрик функций и каррирования.

```js
function createCounter() {
  let count = 0; // эта переменная «замкнута»
  return function () {
    count++;
    return count;
  };
}

const counter = createCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
```

### 10. Что такое callback-функция?

Callback (обратный вызов) — это функция, которая передаётся как аргумент в другую функцию и вызывается позже, обычно после завершения какой-либо операции. Это основной паттерн для обработки асинхронных операций в JavaScript (хотя сейчас чаще используют промисы и async/await). Коллбэки также широко используются в методах массивов: `map`, `filter`, `forEach` и т. д.

```js
function fetchData(callback) {
  setTimeout(() => {
    const data = { name: "React" };
    callback(data); // вызываем callback, когда данные готовы
  }, 1000);
}

fetchData(function (result) {
  console.log(result); // { name: "React" }
});

// Callback в методе массива
[1, 2, 3].map((num) => num * 2); // [2, 4, 6]
```

### 11. Что такое IIFE (Immediately Invoked Function Expression)?

IIFE — это функция, которая **объявляется и сразу же вызывается**. Она создаёт собственную область видимости, что предотвращает загрязнение глобального пространства имён. До появления модулей в ES6 это был основной способ изоляции кода. Сейчас IIFE используется реже, но всё ещё полезно в некоторых сценариях (например, для немедленного выполнения async-кода на верхнем уровне).

```js
(function () {
  const secret = "скрытая переменная";
  console.log(secret); // "скрытая переменная"
})();

// console.log(secret); // ReferenceError — переменная недоступна снаружи

// Со стрелочной функцией
(() => {
  console.log("Выполнено сразу!");
})();
```

### 12. Что такое чистая функция (pure function)?

Чистая функция — это функция, которая: 1) при одинаковых входных данных **всегда** возвращает одинаковый результат; 2) **не имеет побочных эффектов** (не изменяет внешние переменные, не делает запросы, не пишет в DOM и т. д.). Чистые функции делают код предсказуемым, легко тестируемым и отлаживаемым. Reducer в Redux — классический пример чистой функции.

```js
// Чистая функция
function add(a, b) {
  return a + b; // всегда одинаковый результат, нет побочных эффектов
}

// Нечистая функция
let total = 0;
function addToTotal(value) {
  total += value; // побочный эффект — изменяет внешнюю переменную
  return total;
}
```

### Объекты и массивы

### 13. Как скопировать объект? В чём разница между shallow copy и deep copy?

**Shallow copy** (поверхностная копия) копирует только первый уровень свойств. Если объект содержит вложенные объекты, то копируются лишь ссылки на них, а не сами вложенные объекты. **Deep copy** (глубокая копия) рекурсивно копирует все уровни вложенности, создавая полностью независимый клон. Для shallow copy используют `Object.assign()` или spread-оператор. Для deep copy — `structuredClone()` (современный способ) или `JSON.parse(JSON.stringify(...))` (с ограничениями).

```js
const original = { a: 1, nested: { b: 2 } };

// Shallow copy
const shallow = { ...original };
shallow.nested.b = 99;
console.log(original.nested.b); // 99 — вложенный объект изменился!

// Deep copy
const deep = structuredClone(original);
deep.nested.b = 42;
console.log(original.nested.b); // 99 — оригинал не затронут
```

### 14. Какие методы массивов вы знаете? (map, filter, reduce, forEach, find, some, every)

- **`map`** — создаёт новый массив, применяя функцию к каждому элементу.
- **`filter`** — возвращает новый массив из элементов, прошедших проверку.
- **`reduce`** — «сворачивает» массив в одно значение (аккумулятор).
- **`forEach`** — выполняет функцию для каждого элемента, ничего не возвращает.
- **`find`** — возвращает **первый** элемент, удовлетворяющий условию (или `undefined`).
- **`some`** — возвращает `true`, если **хотя бы один** элемент проходит проверку.
- **`every`** — возвращает `true`, если **все** элементы проходят проверку.

```js
const nums = [1, 2, 3, 4, 5];

nums.map((n) => n * 2);        // [2, 4, 6, 8, 10]
nums.filter((n) => n > 3);     // [4, 5]
nums.reduce((sum, n) => sum + n, 0); // 15
nums.find((n) => n > 3);       // 4
nums.some((n) => n > 4);       // true
nums.every((n) => n > 0);      // true
```

### 15. Что такое деструктуризация (destructuring)?

Деструктуризация — это синтаксис, позволяющий «распаковывать» значения из массивов или свойства из объектов в отдельные переменные. Это делает код более лаконичным и читаемым. Можно задавать значения по умолчанию и переименовывать переменные. В React деструктуризация используется повсеместно — для разбора props, значений хуков и т. д.

```js
// Деструктуризация объекта
const user = { name: "Андрей", age: 25, city: "Москва" };
const { name, age, country = "Россия" } = user;

// Деструктуризация массива
const [first, second, ...rest] = [10, 20, 30, 40];
// first = 10, second = 20, rest = [30, 40]

// В React
const [count, setCount] = useState(0);
function Profile({ name, age }) { /* ... */ }
```

### 16. Что такое spread-оператор и rest-параметры?

**Spread-оператор** (`...`) «раскладывает» элементы массива или свойства объекта. Используется для копирования, объединения или передачи аргументов. **Rest-параметры** (`...`) — тот же синтаксис, но в контексте определения функции или деструктуризации он «собирает» оставшиеся элементы в массив. Spread — раскладывает, rest — собирает.

```js
// Spread — раскладывает
const arr1 = [1, 2];
const arr2 = [...arr1, 3, 4]; // [1, 2, 3, 4]

const obj1 = { a: 1 };
const obj2 = { ...obj1, b: 2 }; // { a: 1, b: 2 }

// Rest — собирает
function sum(...numbers) {
  return numbers.reduce((acc, n) => acc + n, 0);
}
sum(1, 2, 3); // 6

const [first, ...others] = [10, 20, 30]; // first=10, others=[20,30]
```

### 17. Как работает оператор опциональной цепочки (?.)?

Оператор опциональной цепочки `?.` позволяет безопасно обращаться к вложенным свойствам объекта. Если значение до `?.` равно `null` или `undefined`, выражение **короткозамыкается** и возвращает `undefined`, вместо того чтобы выбрасывать ошибку `TypeError`. Это значительно упрощает код, избавляя от длинных цепочек проверок на `null`.

```js
const user = {
  name: "Андрей",
  address: {
    city: "Москва",
  },
};

console.log(user.address?.city);    // "Москва"
console.log(user.phone?.number);    // undefined (без ошибки)
console.log(user.getInfo?.());      // undefined (метод не существует, ошибки нет)

// Без ?. пришлось бы писать:
console.log(user.phone && user.phone.number);
```

### Асинхронность

### 18. Что такое Event Loop? Как он работает?

Event Loop (цикл событий) — это механизм, который позволяет JavaScript выполнять асинхронные операции, несмотря на то что JS — однопоточный язык. Он работает так: код выполняется из **стека вызовов (call stack)**. Когда встречается асинхронная операция (таймер, запрос), она передаётся в Web API браузера. Когда операция завершается, её коллбэк помещается в **очередь задач (task queue)**. Event Loop постоянно проверяет: если call stack пуст — берёт задачу из очереди и помещает в стек для выполнения.

```js
console.log("1");           // → в call stack, выполняется сразу

setTimeout(() => {
  console.log("2");         // → в Web API, потом в task queue
}, 0);

console.log("3");           // → в call stack, выполняется сразу

// Вывод: 1, 3, 2
```

### 19. Что такое Promise? Какие у него состояния?

Promise (промис) — это объект, представляющий результат асинхронной операции, который будет доступен в будущем. У промиса три состояния: **pending** (ожидание) — начальное состояние, **fulfilled** (выполнен) — операция завершена успешно, **rejected** (отклонён) — произошла ошибка. Для обработки результата используют `.then()`, а для ошибок — `.catch()`. Промис может перейти из `pending` либо в `fulfilled`, либо в `rejected` — и только один раз.

```js
const promise = new Promise((resolve, reject) => {
  setTimeout(() => {
    const success = true;
    if (success) {
      resolve("Данные получены!");
    } else {
      reject("Ошибка!");
    }
  }, 1000);
});

promise
  .then((data) => console.log(data))   // "Данные получены!"
  .catch((err) => console.error(err))
  .finally(() => console.log("Готово"));
```

### 20. Что такое async/await?

`async/await` — это синтаксический сахар над промисами, делающий асинхронный код похожим на синхронный. Функция, объявленная с `async`, всегда возвращает промис. Внутри неё можно использовать `await`, который «ждёт» разрешения промиса и возвращает результат. Для обработки ошибок используют `try/catch`. Это самый читаемый и удобный способ работы с асинхронностью в современном JavaScript.

```js
async function fetchUser() {
  try {
    const response = await fetch("https://api.example.com/user");
    const data = await response.json();
    console.log(data);
  } catch (error) {
    console.error("Ошибка:", error);
  }
}

fetchUser();
```

### 21. В чём разница между Promise.all, Promise.allSettled, Promise.race и Promise.any?

- **`Promise.all`** — ждёт выполнения **всех** промисов. Если хотя бы один отклонён — весь `Promise.all` отклоняется.
- **`Promise.allSettled`** — ждёт **завершения всех** промисов (неважно, fulfilled или rejected) и возвращает массив с результатом каждого.
- **`Promise.race`** — возвращает результат **первого завершившегося** промиса (будь то fulfilled или rejected).
- **`Promise.any`** — возвращает результат **первого успешно выполненного** промиса. Отклоняется только если все промисы отклонены.

```js
const p1 = Promise.resolve(1);
const p2 = Promise.reject("ошибка");
const p3 = Promise.resolve(3);

Promise.all([p1, p3]).then(console.log);        // [1, 3]
Promise.allSettled([p1, p2]).then(console.log);  // [{status:"fulfilled",value:1}, {status:"rejected",reason:"ошибка"}]
Promise.race([p1, p2]).then(console.log);        // 1 (первый завершившийся)
Promise.any([p2, p3]).then(console.log);         // 3 (первый успешный)
```

### 22. Что такое setTimeout и setInterval?

`setTimeout` — выполняет функцию **один раз** через указанное количество миллисекунд. `setInterval` — выполняет функцию **многократно** с указанным интервалом. Оба возвращают числовой идентификатор, который можно использовать для отмены через `clearTimeout` или `clearInterval`. Важно помнить, что указанная задержка — это минимальная гарантия, а не точное время: коллбэк выполнится не раньше указанного времени, но может и позже (зависит от загруженности event loop).

```js
// Выполнить через 2 секунды
const timerId = setTimeout(() => {
  console.log("Прошло 2 секунды");
}, 2000);

clearTimeout(timerId); // отменить, если нужно

// Выполнять каждую секунду
const intervalId = setInterval(() => {
  console.log("тик");
}, 1000);

clearInterval(intervalId); // остановить
```

### 23. Что такое микротаски и макротаски?

В JavaScript задачи в очереди Event Loop делятся на две категории. **Макротаски** (macrotasks) — это `setTimeout`, `setInterval`, события DOM, `I/O`. **Микротаски** (microtasks) — это `.then()`/`.catch()` промисов, `queueMicrotask()`, `MutationObserver`. Ключевое отличие: после каждой макротаски **все** накопившиеся микротаски выполняются до того, как начнётся следующая макротаска. Поэтому микротаски имеют более высокий приоритет.

```js
console.log("1");                          // синхронно

setTimeout(() => console.log("2"), 0);     // макротаска

Promise.resolve().then(() => console.log("3")); // микротаска

console.log("4");                          // синхронно

// Вывод: 1, 4, 3, 2
// Сначала синхронный код, потом микротаска, потом макротаска
```

### Прототипы и ООП

### 24. Что такое прототипное наследование?

Прототипное наследование — это механизм, при котором объекты могут наследовать свойства и методы от других объектов через цепочку прототипов. У каждого объекта в JavaScript есть скрытое свойство `[[Prototype]]` (доступно через `__proto__` или `Object.getPrototypeOf()`), которое ссылается на другой объект-прототип. Когда мы обращаемся к свойству объекта, JS сначала ищет его в самом объекте, а если не находит — идёт по цепочке прототипов вверх, пока не найдёт или не дойдёт до `null`.

```js
const animal = {
  eat() {
    console.log("Ем...");
  },
};

const cat = Object.create(animal); // cat наследует от animal
cat.meow = function () {
  console.log("Мяу!");
};

cat.meow(); // "Мяу!" — собственный метод
cat.eat();  // "Ем..." — унаследовано от animal
```

### 25. Что такое this? Как определяется контекст вызова?

`this` — это ключевое слово, которое ссылается на объект, в контексте которого выполняется функция. Его значение определяется **в момент вызова**, а не объявления функции. Основные правила: 1) при вызове как метод объекта (`obj.method()`) — `this` указывает на `obj`; 2) при обычном вызове (`func()`) — `this` равен `undefined` (в strict mode) или `window` (в нестрогом); 3) в стрелочных функциях — `this` берётся из окружающего лексического контекста; 4) при использовании `call/apply/bind` — `this` задаётся явно; 5) при вызове с `new` — `this` указывает на новый создаваемый объект.

```js
const user = {
  name: "Андрей",
  greet() {
    console.log(this.name); // "Андрей" — this = user
  },
  delayedGreet() {
    setTimeout(() => {
      console.log(this.name); // "Андрей" — стрелочная берёт this из delayedGreet
    }, 100);
  },
};
```

### 26. Что делают методы call, apply, bind?

Все три метода позволяют явно задать значение `this` для функции. **`call`** вызывает функцию с указанным `this` и аргументами, переданными через запятую. **`apply`** делает то же самое, но аргументы передаются массивом. **`bind`** **не вызывает** функцию сразу, а возвращает **новую функцию** с привязанным `this` (и, опционально, частично применёнными аргументами).

```js
function greet(greeting, punctuation) {
  console.log(`${greeting}, ${this.name}${punctuation}`);
}

const user = { name: "Андрей" };

greet.call(user, "Привет", "!");    // "Привет, Андрей!"
greet.apply(user, ["Привет", "!"]); // "Привет, Андрей!"

const boundGreet = greet.bind(user, "Привет");
boundGreet(".");                     // "Привет, Андрей."
```

### 27. Что такое классы в JavaScript? Чем они отличаются от функций-конструкторов?

Классы в JavaScript — это синтаксический сахар над прототипным наследованием, появившийся в ES6. Они предоставляют более привычный и читаемый синтаксис для создания объектов и настройки наследования. Под капотом классы всё так же используют прототипы. Отличия от функций-конструкторов: классы обязательно вызываются с `new`, их методы по умолчанию неперечислимы, они работают в strict mode, и они не поднимаются (hoisting).

```js
// Функция-конструктор (старый способ)
function Animal(name) {
  this.name = name;
}
Animal.prototype.speak = function () {
  console.log(`${this.name} говорит.`);
};

// Класс (современный способ)
class Animal {
  constructor(name) {
    this.name = name;
  }
  speak() {
    console.log(`${this.name} говорит.`);
  }
}

const dog = new Animal("Бобик");
dog.speak(); // "Бобик говорит."
```

### ES6+ возможности

### 28. Что такое шаблонные строки (template literals)?

Шаблонные строки — это строки, заключённые в обратные кавычки (`` ` ``), появившиеся в ES6. Они позволяют: 1) встраивать выражения через `${...}` (интерполяция); 2) создавать многострочные строки без символов `\n`; 3) использовать тегированные шаблоны (tagged templates) для продвинутой обработки. Это удобнее, чем конкатенация через `+`.

```js
const name = "Андрей";
const age = 25;

// Интерполяция
const greeting = `Привет, ${name}! Тебе ${age} лет.`;

// Многострочность
const html = `
  <div>
    <h1>${name}</h1>
    <p>Возраст: ${age}</p>
  </div>
`;

// Выражения внутри ${}
console.log(`2 + 2 = ${2 + 2}`); // "2 + 2 = 4"
```

### 29. Что такое Map и Set? Чем они отличаются от обычных объектов и массивов?

**`Map`** — это коллекция пар «ключ-значение», где ключом может быть **любое значение** (в обычном объекте ключи — только строки и символы). Map сохраняет порядок вставки и имеет удобное свойство `.size`. **`Set`** — это коллекция **уникальных** значений (в отличие от массива, дубликаты автоматически игнорируются). Оба имеют удобные методы `has`, `add`/`set`, `delete`, `forEach` и являются итерируемыми.

```js
// Map
const map = new Map();
map.set({ id: 1 }, "объект как ключ");
map.set(42, "число как ключ");
console.log(map.size); // 2

// Set
const set = new Set([1, 2, 2, 3, 3]);
console.log([...set]); // [1, 2, 3] — только уникальные
set.has(2); // true
set.add(4);
```

### 30. Что такое символы (Symbol)?

`Symbol` — это примитивный тип данных, введённый в ES6, который создаёт **гарантированно уникальный** идентификатор. Каждый вызов `Symbol()` создаёт новый уникальный символ, даже если описания совпадают. Символы используются в качестве ключей свойств объектов, чтобы избежать конфликтов имён. Также существуют встроенные символы (например, `Symbol.iterator`), которые позволяют кастомизировать поведение объектов.

```js
const id1 = Symbol("id");
const id2 = Symbol("id");
console.log(id1 === id2); // false — каждый символ уникален

const user = {
  name: "Андрей",
  [id1]: 123, // символ как ключ свойства
};

console.log(user[id1]); // 123
console.log(Object.keys(user)); // ["name"] — символьные ключи не видны
```

### 31. Что такое модули в JavaScript? В чём разница между import/export и require/module.exports?

Модули позволяют разбивать код на отдельные файлы с чётко определённым интерфейсом (что экспортируется и импортируется). **ES Modules** (`import/export`) — современный стандарт, поддерживается браузерами и сборщиками. Они статические (анализируются до выполнения), поддерживают tree-shaking. **CommonJS** (`require/module.exports`) — система модулей Node.js. Они динамические (выполняются в рантайме) и загружаются синхронно. Сейчас в новых проектах рекомендуется использовать ES Modules.

```js
// ES Modules
// math.js
export const PI = 3.14;
export function add(a, b) { return a + b; }
export default function multiply(a, b) { return a * b; }

// app.js
import multiply, { PI, add } from "./math.js";

// CommonJS
// math.js
module.exports = { PI: 3.14, add: (a, b) => a + b };

// app.js
const { PI, add } = require("./math.js");
```

## HTML / CSS

### 32. Что такое семантическая вёрстка? Зачем она нужна?

Семантическая вёрстка — это использование HTML-элементов **по их смысловому назначению**, а не только для визуального оформления. Вместо того чтобы использовать `<div>` для всего, мы используем `<header>`, `<nav>`, `<main>`, `<article>`, `<section>`, `<footer>` и другие семантические теги. Это важно потому что: 1) улучшает **доступность** (accessibility) — программы чтения с экрана понимают структуру страницы; 2) улучшает **SEO** — поисковые роботы лучше индексируют контент; 3) повышает **читаемость** кода для разработчиков.

```html
<!-- Плохо — всё на div -->
<div class="header">...</div>
<div class="nav">...</div>

<!-- Хорошо — семантические теги -->
<header>...</header>
<nav>...</nav>
<main>
  <article>...</article>
</main>
<footer>...</footer>
```

### 33. Что такое блочная модель (box model)?

Box model (блочная модель) — это концепция, определяющая, как рассчитывается размер и пространство вокруг каждого HTML-элемента. Каждый элемент представляет собой «коробку», состоящую из четырёх слоёв (изнутри наружу): **content** (содержимое), **padding** (внутренний отступ), **border** (граница), **margin** (внешний отступ). По умолчанию (`box-sizing: content-box`) свойства `width` и `height` задают размер только содержимого, а padding и border добавляются сверху. С `box-sizing: border-box` — width/height включают padding и border, что гораздо удобнее.

```css
/* Рекомендуемая практика */
*,
*::before,
*::after {
  box-sizing: border-box;
}

.box {
  width: 200px;     /* с border-box: итого 200px включая padding и border */
  padding: 20px;
  border: 2px solid black;
  margin: 10px;
}
```

### 34. В чём разница между display: block, inline, inline-block, flex, grid?

- **`block`** — элемент занимает всю ширину, начинается с новой строки. Можно задать width/height, margin/padding. Примеры: `<div>`, `<p>`, `<h1>`.
- **`inline`** — элемент идёт в потоке текста, занимает только столько места, сколько нужно. Нельзя задать width/height, вертикальные margin не работают. Примеры: `<span>`, `<a>`.
- **`inline-block`** — как `inline` (в потоке текста), но можно задать width/height и все отступы.
- **`flex`** — превращает элемент во flex-контейнер для одномерного расположения дочерних элементов (по строке или колонке).
- **`grid`** — превращает элемент в grid-контейнер для двумерной сетки (строки и колонки одновременно).

```css
.container-flex {
  display: flex;
  justify-content: center;
  align-items: center;
}

.container-grid {
  display: grid;
  grid-template-columns: 1fr 1fr 1fr;
  gap: 10px;
}
```

### 35. Что такое позиционирование (static, relative, absolute, fixed, sticky)?

- **`static`** — значение по умолчанию. Элемент в обычном потоке документа, свойства top/right/bottom/left не работают.
- **`relative`** — элемент остаётся в потоке, но может быть сдвинут относительно **своей нормальной позиции** через top/left и т. д.
- **`absolute`** — элемент **извлекается из потока** и позиционируется относительно ближайшего родителя с `position` не `static`.
- **`fixed`** — элемент извлекается из потока и позиционируется относительно **окна браузера**. Остаётся на месте при прокрутке.
- **`sticky`** — гибрид relative и fixed: ведёт себя как relative, пока не достигает заданного порога при прокрутке, после чего «приклеивается».

```css
.tooltip {
  position: absolute;
  top: 100%;
  left: 0;
}

.navbar {
  position: sticky;
  top: 0;
}
```

### 36. Что такое адаптивная вёрстка? Что такое медиа-запросы?

Адаптивная вёрстка (responsive design) — это подход, при котором веб-страница корректно отображается на устройствах с разным размером экрана: от мобильных телефонов до широких мониторов. **Медиа-запросы** (`@media`) — основной инструмент адаптивной вёрстки. Они позволяют применять CSS-правила только при выполнении определённых условий (ширина экрана, ориентация, тип устройства и т. д.). Популярный подход — **mobile-first**: сначала пишутся стили для мобильных, затем через `@media (min-width: ...)` добавляются стили для больших экранов.

```css
/* Mobile first */
.container {
  padding: 10px;
}

@media (min-width: 768px) {
  .container {
    padding: 20px;
    max-width: 720px;
  }
}

@media (min-width: 1200px) {
  .container {
    max-width: 1140px;
  }
}
```

### 37. Что такое Flexbox? Назовите основные свойства.

Flexbox (Flexible Box Layout) — это CSS-модуль для **одномерного** расположения элементов в строку или колонку. Основные свойства контейнера (`display: flex`): `flex-direction` (направление: row/column), `justify-content` (выравнивание по главной оси), `align-items` (выравнивание по поперечной оси), `flex-wrap` (перенос строк), `gap` (отступы между элементами). Свойства дочерних элементов: `flex-grow` (растяжение), `flex-shrink` (сжатие), `flex-basis` (базовый размер), `align-self` (индивидуальное выравнивание).

```css
.container {
  display: flex;
  flex-direction: row;
  justify-content: space-between;
  align-items: center;
  flex-wrap: wrap;
  gap: 16px;
}

.item {
  flex: 1; /* сокращение для flex-grow: 1, flex-shrink: 1, flex-basis: 0% */
}
```

### 38. Что такое CSS Grid? Чем он отличается от Flexbox?

CSS Grid — это модуль для создания **двумерных** сеток (строки и столбцы одновременно). В отличие от Flexbox, который работает в одном измерении (либо строка, либо колонка), Grid позволяет контролировать расположение элементов сразу по обеим осям. Grid идеален для общей компоновки страницы (layout), а Flexbox — для выравнивания элементов внутри секции. На практике их часто комбинируют.

```css
.layout {
  display: grid;
  grid-template-columns: 200px 1fr 200px; /* 3 колонки */
  grid-template-rows: auto 1fr auto;
  gap: 20px;
  min-height: 100vh;
}

.header  { grid-column: 1 / -1; } /* на всю ширину */
.sidebar { grid-column: 1; }
.content { grid-column: 2; }
.aside   { grid-column: 3; }
.footer  { grid-column: 1 / -1; }
```

### 39. Что такое специфичность (specificity) CSS-селекторов?

Специфичность — это механизм, определяющий, какое CSS-правило применяется к элементу при конфликте. Она рассчитывается по формуле с четырьмя уровнями (от высшего к низшему): 1) **inline-стили** (`style="..."`) — наивысший приоритет; 2) **id-селекторы** (`#id`); 3) **классы, атрибуты, псевдоклассы** (`.class`, `[attr]`, `:hover`); 4) **теги и псевдоэлементы** (`div`, `::before`). Чем выше специфичность, тем приоритетнее правило. `!important` перебивает всё (но использовать его не рекомендуется).

```css
p { color: black; }                   /* специфичность: 0-0-1 */
.text { color: blue; }                /* специфичность: 0-1-0 */
#title { color: red; }                /* специфичность: 1-0-0 */
p.text { color: green; }              /* специфичность: 0-1-1 */

/* #title победит, потому что у него самая высокая специфичность */
```

### 40. Что такое БЭМ (BEM)?

БЭМ (Блок, Элемент, Модификатор) — это методология именования CSS-классов, разработанная в Яндексе. **Блок** — это самостоятельный компонент (`header`, `menu`, `card`). **Элемент** — часть блока, которая не имеет смысла вне его (разделитель `__`). **Модификатор** — вариация блока или элемента (разделитель `--`). БЭМ помогает избежать конфликтов имён, делает CSS предсказуемым и облегчает поддержку кода в больших проектах.

```css
/* Блок */
.card { }

/* Элемент блока */
.card__title { }
.card__image { }
.card__description { }

/* Модификатор */
.card--highlighted { }
.card__title--large { }
```

### 41. Что такое CSS-препроцессоры (Sass, Less)?

CSS-препроцессоры — это инструменты, которые расширяют возможности обычного CSS и компилируются в обычный CSS. **Sass** (SCSS) и **Less** — самые популярные из них. Они добавляют: **переменные**, **вложенность** селекторов, **миксины** (примеси — переиспользуемые блоки стилей), **функции**, **наследование** (`@extend`), **математические операции** и разбиение на файлы (`@import`/`@use`). Это делает CSS более структурированным и поддерживаемым. В современном CSS уже есть нативные переменные (`--var`) и вложенность, что уменьшает потребность в препроцессорах.

```scss
// Sass (SCSS)
$primary-color: #3498db;
$border-radius: 8px;

@mixin flex-center {
  display: flex;
  justify-content: center;
  align-items: center;
}

.card {
  background: white;
  border-radius: $border-radius;

  &__title {
    color: $primary-color;
  }

  &:hover {
    @include flex-center;
  }
}
```

## React (основы)

### 42. Что такое React? Зачем он нужен?

React — это JavaScript-библиотека для построения пользовательских интерфейсов, разработанная Facebook (Meta). Она позволяет создавать UI из переиспользуемых **компонентов**, каждый из которых управляет своим состоянием. React нужен для того, чтобы эффективно обновлять и рендерить интерфейс при изменении данных, используя **Virtual DOM** для оптимизации. Он особенно хорош для создания **SPA** (одностраничных приложений) со сложным динамическим интерфейсом.

### 43. Что такое Virtual DOM? Как он работает?

Virtual DOM — это легковесная копия реального DOM, которую React хранит в памяти. Когда состояние компонента меняется, React создаёт новый Virtual DOM, сравнивает его с предыдущей версией (процесс называется **diffing**), находит минимальный набор изменений и применяет **только их** к реальному DOM. Прямые манипуляции с DOM медленны, поэтому этот подход значительно повышает производительность. Разработчик описывает, **как должен выглядеть UI**, а React сам решает, что нужно обновить.

### 44. Что такое JSX? Чем он отличается от HTML?

JSX (JavaScript XML) — это расширение синтаксиса JavaScript, позволяющее писать разметку прямо в JavaScript-коде. Он похож на HTML, но имеет отличия: вместо `class` используется `className`, вместо `for` — `htmlFor`, все теги должны быть закрыты (включая `<img />`), стили задаются как объект (`style={{ color: "red" }}`), можно встраивать JS-выражения через `{}`. JSX компилируется (Babel) в вызовы `React.createElement()`.

```jsx
// JSX
const element = (
  <div className="container">
    <h1 style={{ color: "blue" }}>Привет, {name}!</h1>
    <img src={avatar} alt="аватар" />
  </div>
);

// Компилируется в:
const element = React.createElement(
  "div",
  { className: "container" },
  React.createElement("h1", { style: { color: "blue" } }, "Привет, ", name, "!"),
  React.createElement("img", { src: avatar, alt: "аватар" })
);
```

### 45. Что такое компонент в React?

Компонент — это **независимый, переиспользуемый блок UI**, который принимает входные данные (props) и возвращает React-элементы, описывающие, что должно отображаться на экране. Компоненты бывают функциональными (функция, возвращающая JSX) и классовыми (класс с методом `render()`). Каждый компонент может иметь своё собственное состояние (state), обрабатывать события и содержать другие компоненты. Это основная единица построения интерфейса в React.

```jsx
// Функциональный компонент
function Greeting({ name }) {
  return <h1>Привет, {name}!</h1>;
}

// Использование
<Greeting name="Андрей" />
```

### 46. В чём разница между функциональными и классовыми компонентами?

**Функциональные компоненты** — это обычные функции, которые принимают `props` и возвращают JSX. С появлением хуков (React 16.8) они получили возможность работать с состоянием и жизненным циклом. **Классовые компоненты** — это ES6-классы, наследующие от `React.Component`, с методом `render()` и методами жизненного цикла (`componentDidMount`, `componentDidUpdate` и т. д.). Сейчас функциональные компоненты — стандарт и рекомендуемый подход. Они проще, лаконичнее и легче тестируются.

```jsx
// Функциональный (современный подход)
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

// Классовый (устаревший подход)
class Counter extends React.Component {
  state = { count: 0 };
  render() {
    return (
      <button onClick={() => this.setState({ count: this.state.count + 1 })}>
        {this.state.count}
      </button>
    );
  }
}
```

### 47. Что такое React Fiber?

React Fiber — это движок согласования (reconciliation engine), полностью переписанный в React 16. Его главное преимущество — **инкрементальный рендеринг**: возможность разбивать работу по обновлению UI на небольшие «единицы работы» (fibers) и распределять их по нескольким кадрам. Это позволяет React приостанавливать, приоритизировать и возобновлять работу, не блокируя основной поток. Благодаря Fiber анимации и взаимодействия с пользователем остаются плавными даже при тяжёлых обновлениях. Fiber также является основой для Concurrent Mode и Suspense.

### 48. Что такое reconciliation (согласование)?

Reconciliation — это процесс, в ходе которого React сравнивает новое дерево Virtual DOM с предыдущим и определяет, какие минимальные изменения нужно внести в реальный DOM. React использует эвристический алгоритм O(n) с двумя ключевыми допущениями: 1) элементы разных типов создают разные деревья; 2) разработчик может указать стабильные элементы через атрибут `key`. Именно благодаря reconciliation React эффективно обновляет интерфейс, а не перерисовывает всё заново.

### Props и State

### 49. Что такое props? Можно ли их изменять?

Props (properties) — это данные, которые передаются компоненту от родителя, подобно аргументам функции. Props являются **только для чтения** (read-only) — компонент **не должен изменять** свои props. Это фундаментальный принцип React: данные текут сверху вниз (от родителя к ребёнку). Если нужно изменить данные, родитель должен передать callback-функцию, через которую ребёнок может сообщить о необходимости изменений.

```jsx
function Welcome({ name, onLogout }) {
  // name — данные от родителя (только чтение)
  // onLogout — callback для обратной связи
  return (
    <div>
      <h1>Привет, {name}!</h1>
      <button onClick={onLogout}>Выйти</button>
    </div>
  );
}

// Родитель передаёт props
<Welcome name="Андрей" onLogout={() => console.log("Выход")} />
```

### 50. Что такое state? Чем он отличается от props?

State (состояние) — это **внутренние данные компонента**, которые он может изменять самостоятельно. Когда state меняется, компонент **перерисовывается**. Главные отличия от props: 1) state принадлежит самому компоненту, props передаются извне; 2) state можно изменять (через `setState`/setter-хуков), props — нельзя; 3) state инициализируется внутри компонента, props — родителем. Props — это «конфигурация» компонента, state — его «память».

```jsx
function Counter({ initialCount }) {
  // initialCount — props (от родителя, только для чтения)
  // count — state (собственное состояние, можно менять)
  const [count, setCount] = useState(initialCount);

  return <button onClick={() => setCount(count + 1)}>Счёт: {count}</button>;
}
```

### 51. Почему нельзя изменять state напрямую?

React не отслеживает прямые мутации state — он узнаёт о необходимости перерисовки **только** через вызов `setState` (в классовых компонентах) или setter-функцию из `useState` (в функциональных). Если изменить state напрямую (`this.state.count = 5`), React не узнает об изменении и **не перерисует** компонент. Кроме того, React может **батчить** (группировать) несколько обновлений state для оптимизации, что требует иммутабельного подхода.

```jsx
// НЕПРАВИЛЬНО — мутация напрямую
const [user, setUser] = useState({ name: "Андрей", age: 25 });
user.age = 26; // React не узнает об изменении — ререндера не будет

// ПРАВИЛЬНО — создаём новый объект
setUser({ ...user, age: 26 }); // React видит новую ссылку → перерисовка
```

### 52. Что такое lifting state up (подъём состояния)?

Подъём состояния — это паттерн, при котором общее состояние **переносится вверх** к ближайшему общему родителю компонентов, которые используют это состояние. Вместо того чтобы каждый компонент хранил своё состояние, родитель хранит единственный источник истины и передаёт данные вниз через props. Это нужно, когда нескольким компонентам требуется синхронизировать или разделять одни и те же данные.

```jsx
function Parent() {
  const [text, setText] = useState(""); // состояние поднято в родителя

  return (
    <div>
      <Input value={text} onChange={setText} />
      <Preview text={text} />
    </div>
  );
}

function Input({ value, onChange }) {
  return <input value={value} onChange={(e) => onChange(e.target.value)} />;
}

function Preview({ text }) {
  return <p>Предпросмотр: {text}</p>;
}
```

### 53. Что такое однонаправленный поток данных (one-way data flow)?

Однонаправленный поток данных означает, что в React данные всегда передаются **сверху вниз**: от родительских компонентов к дочерним через props. Дочерний компонент не может напрямую изменить данные родителя. Если ребёнку нужно повлиять на родителя, он вызывает **callback-функцию**, переданную через props. Это делает поток данных предсказуемым, упрощает отладку и понимание того, откуда пришли данные.

```jsx
function App() {
  const [items, setItems] = useState([]);

  const addItem = (item) => setItems([...items, item]); // callback

  return (
    <div>
      {/* Данные текут вниз */}
      <ItemList items={items} />
      {/* Callback для обратной связи */}
      <AddItemForm onAdd={addItem} />
    </div>
  );
}
```

### Хуки (Hooks)

### 54. Что такое хуки? Зачем они нужны?

Хуки (Hooks) — это функции, появившиеся в React 16.8, которые позволяют использовать состояние и другие возможности React **в функциональных компонентах** (без написания классов). До хуков для работы со state и жизненным циклом требовались классовые компоненты. Хуки решили проблемы: сложности переиспользования логики между компонентами, «огромных» классов и запутанных методов жизненного цикла. Основные хуки: `useState`, `useEffect`, `useContext`, `useRef`, `useMemo`, `useCallback`.

### 55. Как работает useState?

`useState` — это хук для добавления **локального состояния** в функциональный компонент. Он принимает **начальное значение** и возвращает массив из двух элементов: текущее значение состояния и функцию для его обновления. При вызове setter-функции компонент **перерисовывается** с новым значением. Начальное значение используется только при первом рендере. Setter может принимать новое значение или функцию, получающую предыдущее значение.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Счёт: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      {/* Функциональное обновление — безопаснее при нескольких обновлениях */}
      <button onClick={() => setCount((prev) => prev + 1)}>+1 (функц.)</button>
    </div>
  );
}
```

### 56. Как работает useEffect? Что такое массив зависимостей?

`useEffect` — это хук для выполнения **побочных эффектов** в функциональных компонентах: запросы к API, подписки, таймеры, работа с DOM. Он принимает функцию-эффект и необязательный **массив зависимостей**. Массив зависимостей определяет, **когда** эффект выполняется: пустой массив `[]` — только при монтировании, с переменными `[a, b]` — при изменении a или b, без массива — после каждого рендера. Функция может возвращать **cleanup-функцию** для очистки (отписка, очистка таймера).

```jsx
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    // Эффект: загрузка данных при изменении userId
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then(setUser);

    // Cleanup-функция (необязательно)
    return () => {
      console.log("Очистка предыдущего эффекта");
    };
  }, [userId]); // Зависимость — перезапускается при изменении userId

  return <div>{user?.name}</div>;
}
```

### 57. В чём разница между useEffect и useLayoutEffect?

Оба хука выполняют побочные эффекты, но в разный момент. **`useEffect`** выполняется **асинхронно, после** того как браузер отрисовал (paint) обновлённый экран. Это не блокирует отрисовку и подходит для большинства случаев (загрузка данных, подписки). **`useLayoutEffect`** выполняется **синхронно, после** обновления DOM, но **до** отрисовки на экране. Он нужен, когда требуется прочитать размеры/позиции элементов или синхронно изменить DOM до того, как пользователь увидит результат (чтобы избежать мерцания).

```jsx
// useEffect — для большинства случаев
useEffect(() => {
  fetchData();
}, []);

// useLayoutEffect — для работы с размерами DOM
useLayoutEffect(() => {
  const height = ref.current.getBoundingClientRect().height;
  setElementHeight(height); // обновление до отрисовки — нет мерцания
}, []);
```

### 58. Как работает useRef? Для чего его используют?

`useRef` возвращает мутабельный объект `{ current: initialValue }`, который **сохраняется между рендерами** и **не вызывает перерисовку** при изменении. Два основных применения: 1) **доступ к DOM-элементам** — через атрибут `ref` на JSX-элементе; 2) **хранение любого мутабельного значения**, которое не должно вызывать ререндер (id таймера, предыдущее значение, флаг и т. д.).

```jsx
function TextInput() {
  const inputRef = useRef(null);
  const renderCount = useRef(0);

  renderCount.current++; // не вызывает ререндер

  const focusInput = () => {
    inputRef.current.focus(); // доступ к DOM-элементу
  };

  return (
    <div>
      <input ref={inputRef} />
      <button onClick={focusInput}>Фокус на поле</button>
      <p>Рендеров: {renderCount.current}</p>
    </div>
  );
}
```

### 59. Как работает useMemo? Когда его стоит использовать?

`useMemo` — это хук для **мемоизации вычислений**. Он принимает функцию и массив зависимостей, выполняет функцию и кэширует результат. При последующих рендерах, если зависимости не изменились, возвращается кэшированное значение без повторного вычисления. Стоит использовать, когда: вычисление **действительно тяжёлое** (фильтрация больших массивов, сложные расчёты); или нужно сохранить **стабильную ссылку** на объект/массив, чтобы избежать ненужных ререндеров дочерних компонентов. Не стоит оборачивать в useMemo всё подряд — это добавляет сложность без пользы.

```jsx
function ProductList({ products, query }) {
  const filteredProducts = useMemo(() => {
    console.log("Фильтрация...");
    return products.filter((p) =>
      p.name.toLowerCase().includes(query.toLowerCase())
    );
  }, [products, query]); // пересчитывается только при изменении products или query

  return (
    <ul>
      {filteredProducts.map((p) => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  );
}
```

### 60. Как работает useCallback? Чем он отличается от useMemo?

`useCallback` мемоизирует **саму функцию**, а `useMemo` — **результат вычисления**. `useCallback(fn, deps)` возвращает ту же самую ссылку на функцию `fn` между рендерами, пока зависимости не изменились. Это полезно, когда вы передаёте callback в дочерний компонент, обёрнутый в `React.memo` — без `useCallback` функция создаётся заново каждый рендер и ребёнок будет перерисовываться. По сути, `useCallback(fn, deps)` эквивалентен `useMemo(() => fn, deps)`.

```jsx
function Parent() {
  const [count, setCount] = useState(0);

  // Без useCallback — handleClick создаётся заново каждый рендер
  // С useCallback — та же ссылка, пока deps не изменятся
  const handleClick = useCallback(() => {
    console.log("Клик!");
  }, []); // пустой массив — функция не зависит от внешних значений

  return (
    <div>
      <p>{count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <ExpensiveChild onClick={handleClick} />
    </div>
  );
}

const ExpensiveChild = React.memo(({ onClick }) => {
  console.log("Рендер ребёнка");
  return <button onClick={onClick}>Дочерняя кнопка</button>;
});
```

### 61. Как работает useContext?

`useContext` — это хук для доступа к значению **React Context** без необходимости использовать `<Context.Consumer>`. Сначала создаётся контекст через `createContext()`, затем провайдер (`<Context.Provider value={...}>`) оборачивает дерево компонентов, а любой вложенный компонент может получить значение через `useContext(Context)`. Это удобно для передачи данных «сквозь» дерево компонентов, минуя промежуточные уровни (без prop drilling).

```jsx
const ThemeContext = createContext("light");

function App() {
  const [theme, setTheme] = useState("light");

  return (
    <ThemeContext.Provider value={theme}>
      <Header />
      <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
        Сменить тему
      </button>
    </ThemeContext.Provider>
  );
}

function Header() {
  const theme = useContext(ThemeContext); // получаем значение из контекста
  return <header className={theme}>Шапка ({theme})</header>;
}
```

### 62. Какие правила хуков (Rules of Hooks) вы знаете?

Существует два основных правила:

1. **Вызывайте хуки только на верхнем уровне** — нельзя использовать хуки внутри условий, циклов или вложенных функций. React полагается на **порядок вызова** хуков между рендерами, и он должен быть одинаковым.

2. **Вызывайте хуки только из React-функций** — хуки можно вызывать только в функциональных компонентах или в пользовательских хуках (custom hooks, имя начинается с `use`). Нельзя вызывать их из обычных JS-функций.

```jsx
// НЕПРАВИЛЬНО
function Component({ show }) {
  if (show) {
    const [value, setValue] = useState(""); // нельзя в условии!
  }
}

// ПРАВИЛЬНО
function Component({ show }) {
  const [value, setValue] = useState("");
  if (!show) return null; // условие после хуков
}
```

### Обработка событий и формы

### 63. Как обрабатываются события в React? Чем это отличается от нативного DOM?

В React события обрабатываются через JSX-атрибуты в camelCase: `onClick`, `onChange`, `onSubmit` и т. д. В обработчик передаётся **функция** (не строка, как в HTML). Главные отличия от нативного DOM: 1) React использует **SyntheticEvent** — кроссбраузерную обёртку вокруг нативного события; 2) события делегируются на корневой элемент (event delegation), а не на каждый элемент отдельно; 3) для предотвращения поведения по умолчанию нужно вызвать `e.preventDefault()`, а не вернуть `false`.

```jsx
// React
function Form() {
  const handleSubmit = (e) => {
    e.preventDefault(); // предотвращаем перезагрузку страницы
    console.log("Отправлено!");
  };

  return (
    <form onSubmit={handleSubmit}>
      <button type="submit">Отправить</button>
    </form>
  );
}

// В HTML было бы:
// <button onclick="handleClick()">Клик</button>
// В React:
// <button onClick={handleClick}>Клик</button>
```

### 64. Что такое SyntheticEvent?

SyntheticEvent — это кроссбраузерная обёртка React над нативным событием браузера. Она имеет тот же интерфейс, что и нативное событие (`stopPropagation()`, `preventDefault()`, `target`, `currentTarget`), но работает одинаково во всех браузерах. Доступ к нативному событию можно получить через свойство `e.nativeEvent`. SyntheticEvent пулируется (в старых версиях React) для производительности, но начиная с React 17 пулинг был убран, и объект события доступен асинхронно.

```jsx
function Input() {
  const handleChange = (e) => {
    // e — SyntheticEvent
    console.log(e.target.value);    // значение поля
    console.log(e.nativeEvent);     // нативное событие браузера
    console.log(e.type);            // "change"
  };

  return <input onChange={handleChange} />;
}
```

### 65. Что такое контролируемые (controlled) и неконтролируемые (uncontrolled) компоненты?

**Контролируемый компонент** — это элемент формы, значение которого управляется **React-состоянием**. Значение задаётся через `value`, а изменения обрабатываются через `onChange`. React является единственным «источником истины». **Неконтролируемый компонент** хранит значение в самом DOM, а для доступа к нему используется `useRef`. Контролируемые компоненты рекомендуются в большинстве случаев, так как дают полный контроль над данными формы.

```jsx
// Контролируемый
function ControlledInput() {
  const [value, setValue] = useState("");
  return <input value={value} onChange={(e) => setValue(e.target.value)} />;
}

// Неконтролируемый
function UncontrolledInput() {
  const inputRef = useRef(null);
  const handleSubmit = () => {
    console.log(inputRef.current.value); // читаем из DOM
  };
  return <input ref={inputRef} defaultValue="начальное значение" />;
}
```

### 66. Как работать с формами в React?

Работа с формами в React обычно строится на контролируемых компонентах: каждое поле привязано к state, обновляется через `onChange`, а отправка обрабатывается через `onSubmit` с `e.preventDefault()`. Для сложных форм с валидацией часто используют библиотеки: **React Hook Form**, **Formik**. Для нескольких полей удобно хранить все значения в одном объекте state.

```jsx
function LoginForm() {
  const [form, setForm] = useState({ email: "", password: "" });

  const handleChange = (e) => {
    setForm({ ...form, [e.target.name]: e.target.value });
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log("Отправка:", form);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="email" value={form.email} onChange={handleChange} />
      <input
        name="password"
        type="password"
        value={form.password}
        onChange={handleChange}
      />
      <button type="submit">Войти</button>
    </form>
  );
}
```

### Условный рендеринг и списки

### 67. Как реализовать условный рендеринг?

В React есть несколько способов условного рендеринга: 1) **тернарный оператор** (`condition ? <A /> : <B />`); 2) **логическое И** (`condition && <A />` — отрисовка только если условие истинно); 3) **`if/else`** — обычные условия внутри тела функции до `return`; 4) **ранний return** (`if (!data) return <Loading />`). Выбор зависит от ситуации: тернарный — для переключения между двумя вариантами, `&&` — для показа/скрытия одного элемента.

```jsx
function Dashboard({ user, isLoading }) {
  // Ранний return
  if (isLoading) return <Spinner />;
  if (!user) return <p>Пользователь не найден</p>;

  return (
    <div>
      {/* Тернарный оператор */}
      {user.isAdmin ? <AdminPanel /> : <UserPanel />}

      {/* Логическое И */}
      {user.notifications.length > 0 && <NotificationBadge />}
    </div>
  );
}
```

### 68. Зачем нужен атрибут key при рендеринге списков?

Атрибут `key` помогает React **идентифицировать** каждый элемент списка и эффективно определять, какие элементы были добавлены, удалены или изменены при перерисовке. Без `key` React вынужден перерисовывать все элементы списка. С уникальным и стабильным `key` React может **переиспользовать** существующие DOM-элементы и минимизировать дорогие операции с DOM. Key должен быть уникальным среди соседних элементов (не глобально).

```jsx
function TodoList({ todos }) {
  return (
    <ul>
      {todos.map((todo) => (
        // Используем уникальный id как key
        <li key={todo.id}>{todo.text}</li>
      ))}
    </ul>
  );
}
```

### 69. Почему не рекомендуется использовать индекс массива в качестве key?

Индекс массива как `key` создаёт проблемы, когда порядок элементов **изменяется** (добавление, удаление, сортировка, перемещение). Поскольку индексы переназначаются при каждом изменении, React не сможет правильно определить, какой элемент какой, и может: 1) неправильно обновить DOM; 2) потерять state компонентов; 3) показать некорректные данные в полях ввода. Индекс допустим только если: список статический, не перестраивается и элементы не имеют своего state.

```jsx
// ПЛОХО — индекс как key
{items.map((item, index) => (
  <TodoItem key={index} item={item} />
  // При удалении элемента из середины все индексы сдвинутся
  // и React неправильно сопоставит компоненты
))}

// ХОРОШО — уникальный идентификатор
{items.map((item) => (
  <TodoItem key={item.id} item={item} />
))}
```

### Стилизация

### 70. Какие способы стилизации компонентов в React вы знаете?

Основные способы:
1. **Обычный CSS** — подключение `.css` файлов через `import`.
2. **CSS Modules** — файлы `.module.css` со скоупом на уровне компонента.
3. **Inline styles** — объект стилей через атрибут `style`.
4. **CSS-in-JS** — библиотеки вроде `styled-components`, `emotion`.
5. **Утилитарные фреймворки** — Tailwind CSS, UnoCSS.
6. **CSS-препроцессоры** — Sass/Less.

Каждый подход имеет свои плюсы и минусы. CSS Modules и Tailwind — наиболее популярные в современных проектах.

```jsx
// Inline styles
<div style={{ color: "red", fontSize: "16px" }}>Текст</div>

// CSS Modules
import styles from "./Button.module.css";
<button className={styles.primary}>Кнопка</button>

// Tailwind CSS
<button className="bg-blue-500 text-white px-4 py-2 rounded">Кнопка</button>
```

### 71. Что такое CSS Modules?

CSS Modules — это CSS-файлы, в которых все имена классов **автоматически скоупятся** (ограничиваются) на уровне компонента. Файл именуется `Component.module.css` и импортируется как объект. При сборке каждый класс получает уникальное имя (например, `Button_primary_a3b2c`), что полностью исключает конфликты стилей между компонентами. Это встроено в большинство сборщиков (Vite, Webpack) и не требует дополнительной настройки.

```css
/* Button.module.css */
.primary {
  background-color: blue;
  color: white;
  padding: 8px 16px;
}

.disabled {
  opacity: 0.5;
}
```

```jsx
import styles from "./Button.module.css";

function Button({ disabled }) {
  const className = disabled
    ? `${styles.primary} ${styles.disabled}`
    : styles.primary;

  return <button className={className}>Кнопка</button>;
}
// В DOM: <button class="Button_primary_a3b2c">Кнопка</button>
```

### 72. Что такое styled-components / CSS-in-JS?

CSS-in-JS — это подход, при котором стили пишутся **прямо в JavaScript**. `styled-components` — самая популярная библиотека этого подхода. Она позволяет создавать React-компоненты со встроенными стилями, используя тегированные шаблонные строки. Стили автоматически скоупятся, поддерживаются динамические стили на основе props, темизация и вложенность. Минус: добавляет runtime-нагрузку (хотя есть zero-runtime альтернативы вроде `linaria`, `vanilla-extract`).

```jsx
import styled from "styled-components";

const Button = styled.button`
  background-color: ${(props) => (props.primary ? "blue" : "gray")};
  color: white;
  padding: 8px 16px;
  border: none;
  border-radius: 4px;
  cursor: pointer;

  &:hover {
    opacity: 0.8;
  }
`;

function App() {
  return (
    <div>
      <Button primary>Основная</Button>
      <Button>Обычная</Button>
    </div>
  );
}
```

## Инструменты и экосистема

### 73. Что такое npm / yarn / pnpm?

Это **менеджеры пакетов** для JavaScript/Node.js, которые позволяют устанавливать, обновлять и управлять зависимостями проекта. **npm** — стандартный менеджер, поставляется вместе с Node.js. **yarn** — разработан Facebook как альтернатива npm с улучшенной скоростью и надёжностью (lockfile). **pnpm** — экономит дисковое пространство, используя жёсткие ссылки и единое глобальное хранилище пакетов. Все три используют `package.json` для описания зависимостей.

```bash
# npm
npm install react
npm run dev

# yarn
yarn add react
yarn dev

# pnpm
pnpm add react
pnpm dev
```

### 74. Что такое Vite? Чем он отличается от Webpack?

**Vite** — это современный инструмент сборки (build tool), разработанный Эваном Ю (создателем Vue). В режиме разработки он использует **нативные ES-модули** браузера и не бандлит код, что обеспечивает **мгновенный запуск** и быструю горячую перезагрузку (HMR). Для продакшн-сборки использует Rollup. **Webpack** — более старый и мощный бандлер, который собирает весь код в один бандл даже в dev-режиме, что делает его медленнее. Vite значительно проще в настройке и быстрее в разработке.

### 75. Что такое Babel? Зачем он нужен?

Babel — это **транспилятор** (компилятор) JavaScript, который преобразует современный JS-код (ES6+, JSX, TypeScript) в более старую версию, совместимую со старыми браузерами. Он также трансформирует JSX в вызовы `React.createElement()`. Babel работает через **плагины и пресеты** (наборы плагинов): `@babel/preset-env` (для ES6+), `@babel/preset-react` (для JSX), `@babel/preset-typescript` (для TS). В проектах с Vite транспиляция часто выполняется `esbuild` или `SWC`, которые значительно быстрее.

### 76. Что такое ESLint и Prettier?

**ESLint** — это инструмент **статического анализа** (линтер), который находит и сообщает о проблемах в JavaScript/TypeScript коде: ошибки, потенциальные баги, нарушения стиля и лучших практик. **Prettier** — это **форматтер** кода, который автоматически приводит код к единому стилю (отступы, кавычки, переносы). ESLint отвечает за **качество кода**, а Prettier — за **форматирование**. Их обычно используют вместе: ESLint проверяет логику, Prettier форматирует. Это обеспечивает единый стиль кода в команде.

### 77. Что такое package.json? Какие основные поля в нём?

`package.json` — это главный конфигурационный файл Node.js-проекта, который содержит метаданные и описание зависимостей. Основные поля:
- **`name`** — имя проекта.
- **`version`** — версия проекта.
- **`scripts`** — команды для запуска (`dev`, `build`, `test`).
- **`dependencies`** — зависимости для продакшена.
- **`devDependencies`** — зависимости только для разработки.
- **`main`** / **`module`** — точка входа.
- **`type`** — тип модулей (`"module"` для ESM).

```json
{
  "name": "my-react-app",
  "version": "1.0.0",
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "test": "vitest"
  },
  "dependencies": {
    "react": "^18.2.0",
    "react-dom": "^18.2.0"
  },
  "devDependencies": {
    "vite": "^5.0.0"
  }
}
```

### Роутинг

### 78. Что такое React Router? Зачем нужен роутинг в SPA?

**React Router** — это самая популярная библиотека маршрутизации для React. В **SPA** (Single Page Application) при навигации между «страницами» на самом деле загрузка новой HTML-страницы **не происходит** — React просто рендерит разные компоненты в зависимости от URL. React Router управляет этим процессом: он следит за URL, рендерит соответствующие компоненты и позволяет навигироваться между «страницами» без перезагрузки.

```jsx
import { BrowserRouter, Routes, Route, Link } from "react-router-dom";

function App() {
  return (
    <BrowserRouter>
      <nav>
        <Link to="/">Главная</Link>
        <Link to="/about">О нас</Link>
      </nav>

      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
        <Route path="*" element={<NotFound />} />
      </Routes>
    </BrowserRouter>
  );
}
```

### 79. В чём разница между Link и a?

Тег `<a>` выполняет **полную перезагрузку страницы** при навигации — браузер делает новый HTTP-запрос, загружает HTML, перезапускает JavaScript и теряет всё текущее состояние приложения. Компонент `<Link>` из React Router перехватывает клик, **обновляет URL через History API** и рендерит нужный компонент **без перезагрузки страницы**. Состояние приложения сохраняется, переход мгновенный. `<Link>` рендерит `<a>` в DOM, но с перехваченным поведением.

```jsx
// ПЛОХО в SPA — вызывает полную перезагрузку
<a href="/about">О нас</a>

// ХОРОШО — клиентская навигация без перезагрузки
<Link to="/about">О нас</Link>

// NavLink — как Link, но умеет определять активный маршрут
<NavLink to="/about" className={({ isActive }) => isActive ? "active" : ""}>
  О нас
</NavLink>
```

### 80. Что такое динамические маршруты?

Динамические маршруты — это маршруты с **параметрами** (переменными частями URL). Они позволяют одному маршруту обрабатывать множество URL. Параметр обозначается двоеточием в пути (`:id`, `:slug` и т. д.) и доступен внутри компонента через хук `useParams()`. Это нужно, например, для страниц товаров, профилей пользователей или статей.

```jsx
// Определение динамического маршрута
<Route path="/users/:userId" element={<UserProfile />} />
<Route path="/posts/:postId" element={<PostPage />} />

// Компонент, использующий параметр
function UserProfile() {
  const { userId } = useParams();

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then(setUser);
  }, [userId]);

  return <div>Профиль пользователя #{userId}</div>;
}

// URL /users/42 → userId = "42"
```

### 81. Как реализовать защищённые маршруты (protected routes)?

Защищённые маршруты — это маршруты, доступные только авторизованным пользователям (или пользователям с определёнными правами). Реализация: создаём компонент-обёртку, который проверяет статус авторизации и либо рендерит дочерний маршрут, либо перенаправляет на страницу входа. Для перенаправления используется компонент `<Navigate>` из React Router.

```jsx
function ProtectedRoute({ children }) {
  const { isAuthenticated } = useAuth(); // кастомный хук авторизации

  if (!isAuthenticated) {
    return <Navigate to="/login" replace />;
  }

  return children;
}

// Использование
<Routes>
  <Route path="/login" element={<LoginPage />} />
  <Route
    path="/dashboard"
    element={
      <ProtectedRoute>
        <Dashboard />
      </ProtectedRoute>
    }
  />
</Routes>
```

### Управление состоянием

### 82. Когда нужен глобальный стейт-менеджер?

Глобальный стейт-менеджер нужен, когда: 1) **многие компоненты** на разных уровнях дерева нуждаются в одних и тех же данных; 2) **prop drilling** (пробрасывание props через много уровней) становится неудобным; 3) нужна **сложная логика** обновления состояния с предсказуемым потоком данных; 4) нужно **сохранять состояние** при навигации между страницами. Для простых случаев достаточно `useState` + `useContext`. Для сложных — Redux, Zustand, MobX, Jotai и другие библиотеки.

### 83. Что такое Context API? Когда его использовать?

Context API — встроенный в React механизм для передачи данных через дерево компонентов **без prop drilling** (пробрасывания через каждый промежуточный компонент). Состоит из `createContext()`, `<Provider>` и хука `useContext()`. Подходит для: темизации, языковых настроек, данных авторизации — данных, которые редко меняются и нужны многим компонентам. **Не подходит** для часто обновляемых данных, так как при изменении контекста **все** потребители перерисовываются.

```jsx
const AuthContext = createContext(null);

function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const login = (userData) => setUser(userData);
  const logout = () => setUser(null);

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

function useAuth() {
  return useContext(AuthContext);
}

function Profile() {
  const { user, logout } = useAuth();
  return <button onClick={logout}>Выйти, {user.name}</button>;
}
```

### 84. Что такое Redux? Какие основные концепции (store, action, reducer)?

Redux — это библиотека для управления **глобальным состоянием** приложения с предсказуемым потоком данных. Основные концепции:
- **Store** — единое хранилище всего состояния приложения.
- **Action** — простой объект с полем `type`, описывающий, **что произошло**.
- **Reducer** — чистая функция, которая принимает текущее состояние и action, и возвращает **новое** состояние.
- **Dispatch** — метод для отправки action в store.

Поток данных: UI → dispatch(action) → reducer → новый state → обновление UI.

```js
// Action
const increment = { type: "counter/increment" };

// Reducer
function counterReducer(state = { value: 0 }, action) {
  switch (action.type) {
    case "counter/increment":
      return { value: state.value + 1 };
    case "counter/decrement":
      return { value: state.value - 1 };
    default:
      return state;
  }
}

// Store
const store = createStore(counterReducer);
store.dispatch(increment);
console.log(store.getState()); // { value: 1 }
```

### 85. Что такое Redux Toolkit?

Redux Toolkit (RTK) — это **официальный рекомендуемый** способ использования Redux. Он значительно упрощает работу с Redux, устраняя boilerplate-код. Ключевые функции: **`configureStore`** — упрощённая настройка store с включёнными middleware; **`createSlice`** — автоматически генерирует actions и reducer; встроенная поддержка **Immer** (можно «мутировать» состояние, а Immer создаст иммутабельное обновление); **RTK Query** — для работы с API.

```js
import { configureStore, createSlice } from "@reduxjs/toolkit";

const counterSlice = createSlice({
  name: "counter",
  initialState: { value: 0 },
  reducers: {
    increment(state) {
      state.value += 1; // Immer позволяет «мутировать» напрямую
    },
    decrement(state) {
      state.value -= 1;
    },
    incrementByAmount(state, action) {
      state.value += action.payload;
    },
  },
});

export const { increment, decrement, incrementByAmount } = counterSlice.actions;

const store = configureStore({
  reducer: { counter: counterSlice.reducer },
});
```

### Работа с API

### 86. Как делать HTTP-запросы в React? (fetch, axios)

Для HTTP-запросов в React используют встроенный `fetch` API или библиотеку `axios`. **`fetch`** — встроен в браузер, возвращает промис, но не отклоняет промис при HTTP-ошибках (4xx, 5xx) и не парсит JSON автоматически. **`axios`** — популярная библиотека с удобным API: автоматически парсит JSON, поддерживает интерцепторы, отменяет запросы, обрабатывает HTTP-ошибки. Запросы обычно делают в `useEffect` или с помощью библиотек (React Query).

```jsx
// fetch
useEffect(() => {
  fetch("https://api.example.com/data")
    .then((res) => {
      if (!res.ok) throw new Error("Ошибка HTTP");
      return res.json();
    })
    .then(setData)
    .catch(setError);
}, []);

// axios
import axios from "axios";

useEffect(() => {
  axios
    .get("https://api.example.com/data")
    .then((res) => setData(res.data))
    .catch(setError);
}, []);
```

### 87. Где лучше всего делать запросы к API в компоненте?

В функциональных компонентах запросы к API лучше всего делать в хуке **`useEffect`**, так как это побочный эффект. Для начальной загрузки данных используют `useEffect` с пустым массивом зависимостей `[]` (эффект выполнится один раз при монтировании). Если данные зависят от параметра (например, `userId`), его добавляют в массив зависимостей. Ещё лучше — использовать специализированные библиотеки вроде **React Query (TanStack Query)**, которые берут на себя кэширование, повторные попытки, загрузку и ошибки.

```jsx
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [userId]); // перезапрос при смене userId

  if (loading) return <Spinner />;
  if (error) return <p>Ошибка: {error.message}</p>;
  return <div>{user.name}</div>;
}
```

### 88. Что такое React Query (TanStack Query)?

React Query (TanStack Query) — это библиотека для управления **серверным состоянием** в React. Она решает задачи: кэширование запросов, автоматические повторные попытки, фоновое обновление данных, пагинация, «оптимистичные» обновления, отмена запросов. Вместо ручного управления `loading`/`error`/`data` в `useEffect`, React Query предоставляет хук `useQuery` с готовыми состояниями. Это значительно сокращает код и делает работу с API надёжнее.

```jsx
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";

function Users() {
  const { data, isLoading, error } = useQuery({
    queryKey: ["users"],
    queryFn: () => fetch("/api/users").then((res) => res.json()),
  });

  if (isLoading) return <Spinner />;
  if (error) return <p>Ошибка: {error.message}</p>;

  return (
    <ul>
      {data.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

## TypeScript (основы)

### 89. Что такое TypeScript? Зачем он нужен?

TypeScript — это **надмножество JavaScript**, добавляющее статическую типизацию. Код на TypeScript компилируется в обычный JavaScript. Зачем он нужен: 1) **ошибки обнаруживаются на этапе разработки**, а не в рантайме; 2) **автодополнение** и подсказки в IDE становятся значительно лучше; 3) код становится **самодокументируемым** — типы описывают, какие данные ожидаются; 4) **безопаснее рефакторинг** — компилятор сразу покажет, где что-то сломалось. TypeScript стал стандартом в большинстве React-проектов.

```ts
// JavaScript — ошибка обнаружится только в рантайме
function add(a, b) {
  return a + b;
}
add("1", 2); // "12" — неожиданная конкатенация

// TypeScript — ошибка обнаружится при компиляции
function add(a: number, b: number): number {
  return a + b;
}
add("1", 2); // Ошибка: Argument of type 'string' is not assignable to parameter of type 'number'
```

### 90. В чём разница между interface и type?

Оба позволяют описывать формы объектов, но есть различия. **`interface`** — предназначен для описания форм объектов и классов. Поддерживает **расширение** (`extends`) и **слияние** (declaration merging — можно объявить интерфейс дважды, и они объединятся). **`type`** — более универсален: может описывать не только объекты, но и **union**, **intersection**, **примитивы**, **кортежи**. Не поддерживает declaration merging. Рекомендация: используйте `interface` для объектов и API, `type` — для union, intersection и алиасов.

```ts
// interface
interface User {
  name: string;
  age: number;
}
interface Admin extends User {
  role: string;
}

// type
type ID = string | number; // union — нельзя сделать через interface
type Point = { x: number; y: number };
type Response = User & { token: string }; // intersection
```

### 91. Что такое дженерики (generics)?

Дженерики — это **параметризованные типы**, позволяющие создавать компоненты (функции, классы, интерфейсы), которые работают с **разными типами данных**, сохраняя типобезопасность. Это как «переменная для типа». Вместо дублирования кода для каждого типа или использования `any`, дженерик принимает тип как параметр и использует его внутри.

```ts
// Без дженериков — нужны отдельные функции или any
function getFirst(arr: any[]): any {
  return arr[0];
}

// С дженериками — сохраняется типобезопасность
function getFirst<T>(arr: T[]): T {
  return arr[0];
}

const num = getFirst([1, 2, 3]);     // тип: number
const str = getFirst(["a", "b"]);    // тип: string

// В React
interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
}
```

### 92. Что такое enum?

`enum` (перечисление) — это способ определить набор **именованных констант**. Это делает код более читаемым и менее подверженным ошибкам, чем использование «магических строк» или чисел. Бывают числовые (значения — автоинкрементные числа) и строковые enum. Однако многие разработчики предпочитают использовать **union-типы** или `as const` объекты вместо enum, так как enum создают дополнительный код при компиляции.

```ts
// Числовой enum
enum Direction {
  Up,    // 0
  Down,  // 1
  Left,  // 2
  Right, // 3
}

// Строковый enum
enum Status {
  Loading = "LOADING",
  Success = "SUCCESS",
  Error = "ERROR",
}

function handleStatus(status: Status) {
  if (status === Status.Loading) {
    console.log("Загрузка...");
  }
}

// Альтернатива — union type (часто предпочтительнее)
type Status = "loading" | "success" | "error";
```

### 93. Что такое unknown и any? В чём разница?

**`any`** полностью **отключает** проверку типов — с переменной типа `any` можно делать что угодно без ошибок компилятора. Это по сути отказ от преимуществ TypeScript. **`unknown`** — безопасная альтернатива: переменная может содержать любое значение, но **перед использованием** нужно проверить (сузить) её тип. `unknown` заставляет разработчика делать проверки, что предотвращает ошибки.

```ts
// any — опасно, никаких проверок
let a: any = "строка";
a.toFixed(2); // Нет ошибки при компиляции, но упадёт в рантайме!

// unknown — безопасно, требует проверки
let b: unknown = "строка";
// b.toFixed(2); // Ошибка компиляции!

if (typeof b === "string") {
  b.toUpperCase(); // OK — тип сужен до string
}

if (typeof b === "number") {
  b.toFixed(2); // OK — тип сужен до number
}
```

### 94. Как типизировать props компонента в React?

Props типизируются с помощью `interface` или `type`, которые передаются как параметр типа функции-компонента. Можно указать необязательные (`?`), читаемые (`readonly`) свойства, значения по умолчанию и тип `children`. Часто используют `React.FC<Props>` (функциональный компонент), но многие предпочитают явную типизацию аргумента.

```tsx
interface ButtonProps {
  label: string;
  onClick: () => void;
  variant?: "primary" | "secondary"; // необязательный prop
  disabled?: boolean;
  children?: React.ReactNode;
}

// Способ 1 — деструктуризация с типом
function Button({ label, onClick, variant = "primary", disabled }: ButtonProps) {
  return (
    <button onClick={onClick} disabled={disabled} className={variant}>
      {label}
    </button>
  );
}

// Способ 2 — React.FC
const Button: React.FC<ButtonProps> = ({ label, onClick }) => {
  return <button onClick={onClick}>{label}</button>;
};
```

### 95. Как типизировать useState и события?

`useState` чаще всего **автоматически выводит тип** из начального значения. Если начальное значение не определяет тип полностью (например, `null` или сложный объект), тип указывается через дженерик. Для событий React предоставляет типы: `React.ChangeEvent`, `React.MouseEvent`, `React.FormEvent` и др.

```tsx
// useState — автоматический вывод
const [count, setCount] = useState(0); // тип: number

// useState — явный тип
const [user, setUser] = useState<User | null>(null);
const [items, setItems] = useState<string[]>([]);

// Типизация событий
function Form() {
  const [value, setValue] = useState("");

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    setValue(e.target.value);
  };

  const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault();
  };

  const handleClick = (e: React.MouseEvent<HTMLButtonElement>) => {
    console.log("Клик");
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={value} onChange={handleChange} />
      <button onClick={handleClick}>OK</button>
    </form>
  );
}
```

## Git и рабочий процесс

### 96. Что такое Git? Зачем он нужен?

Git — это **распределённая система контроля версий**, которая отслеживает изменения в файлах и позволяет нескольким разработчикам работать над одним проектом. Зачем нужен: 1) **история изменений** — можно посмотреть, кто, когда и что изменил; 2) **откат** — можно вернуться к любой предыдущей версии; 3) **параллельная работа** — разработчики работают в отдельных ветках и объединяют изменения; 4) **резервное копирование** — код хранится на удалённых серверах (GitHub, GitLab).

### 97. Какие основные команды Git вы знаете? (clone, add, commit, push, pull, merge, rebase)

- **`git clone <url>`** — клонирует удалённый репозиторий на локальную машину.
- **`git add <файлы>`** — добавляет изменения в staging area (индекс) для следующего коммита.
- **`git commit -m "сообщение"`** — фиксирует изменения из staging area с описанием.
- **`git push`** — отправляет локальные коммиты на удалённый сервер.
- **`git pull`** — получает изменения с сервера и сливает их с локальной веткой.
- **`git merge <ветка>`** — объединяет указанную ветку с текущей (создаёт merge-коммит).
- **`git rebase <ветка>`** — переносит коммиты текущей ветки на вершину указанной (линейная история).

```bash
git clone https://github.com/user/repo.git
git add .
git commit -m "Добавлена авторизация"
git push origin main
git pull origin main
git merge feature-branch
git rebase main
```

### 98. Что такое ветвление (branching)?

Ветвление — это создание **параллельной линии разработки** (ветки), которая отходит от основной. Каждая ветка — это независимая копия кода, где можно безопасно вносить изменения, не затрагивая основную ветку (`main`/`master`). Типичный рабочий процесс: создать ветку для фичи → внести изменения → протестировать → объединить с основной веткой через merge/rebase. Популярные стратегии: **Git Flow**, **GitHub Flow**, **trunk-based development**.

```bash
git branch feature/login      # создать ветку
git checkout feature/login    # переключиться на ветку
# или одной командой:
git checkout -b feature/login # создать и переключиться

git add . && git commit -m "Реализован логин"
git checkout main
git merge feature/login       # объединить с main
git branch -d feature/login   # удалить ветку
```

### 99. Что такое pull request (merge request)?

Pull request (PR) — в GitLab называется merge request (MR) — это **запрос на включение** изменений из одной ветки в другую через веб-интерфейс (GitHub, GitLab, Bitbucket). PR позволяет: 1) провести **код-ревью** — другие разработчики просматривают изменения и оставляют комментарии; 2) запустить **автоматические проверки** (тесты, линтер, CI/CD); 3) обсудить реализацию перед слиянием. Это ключевой инструмент командной работы, обеспечивающий качество кода.

### 100. Как разрешать конфликты при слиянии?

Конфликты возникают, когда **одни и те же строки файла** были изменены в разных ветках. Git помечает конфликтные участки специальными маркерами (`<<<<<<<`, `=======`, `>>>>>>>`). Для разрешения нужно: 1) открыть конфликтный файл; 2) выбрать, какие изменения оставить (свои, чужие, или скомбинировать); 3) удалить маркеры конфликта; 4) сохранить файл, выполнить `git add` и `git commit`. Современные IDE (VS Code) имеют удобные визуальные инструменты для разрешения конфликтов.

```
<<<<<<< HEAD
const title = "Мой вариант";
=======
const title = "Вариант коллеги";
>>>>>>> feature-branch

// Разрешаем конфликт — выбираем нужный вариант:
const title = "Финальный вариант";
```

```bash
git add .
git commit -m "Разрешён конфликт слияния"
```

## Тестирование

### 101. Какие виды тестирования вы знаете? (unit, integration, e2e)

- **Unit-тесты** (модульные) — тестируют **отдельные функции или компоненты** изолированно. Самые быстрые и простые. Пример: проверка работы утилитной функции.
- **Integration-тесты** (интеграционные) — тестируют **взаимодействие** нескольких модулей/компонентов вместе. Пример: проверка формы с валидацией и отправкой.
- **E2E-тесты** (end-to-end, сквозные) — тестируют **приложение целиком** в браузере, имитируя действия реального пользователя. Самые медленные, но ближе к реальности. Инструменты: Cypress, Playwright.

«Пирамида тестирования» рекомендует: много unit-тестов, меньше интеграционных, ещё меньше e2e.

### 102. Что такое Jest?

Jest — это популярный **фреймворк для тестирования JavaScript**, разработанный Meta. Он работает «из коробки» и включает: test runner (запуск тестов), assertion library (проверки — `expect`, `toBe`, `toEqual`), моки и шпионы (`jest.fn()`, `jest.mock()`), покрытие кода (coverage), snapshot-тестирование. Jest быстрый (параллельный запуск), не требует сложной настройки и широко используется в React-проектах. Альтернатива — **Vitest** (совместим с Jest API, но быстрее с Vite).

```js
// sum.js
export function sum(a, b) {
  return a + b;
}

// sum.test.js
import { sum } from "./sum";

describe("sum", () => {
  test("складывает 1 + 2 = 3", () => {
    expect(sum(1, 2)).toBe(3);
  });

  test("складывает отрицательные числа", () => {
    expect(sum(-1, -2)).toBe(-3);
  });
});
```

### 103. Что такое React Testing Library?

React Testing Library (RTL) — это библиотека для тестирования React-компонентов, основанная на философии: **тестируйте поведение, а не реализацию**. Она рендерит компоненты в виртуальный DOM и предоставляет методы для поиска элементов так, как их видит пользователь: по тексту, роли, label, placeholder. В отличие от Enzyme (устаревшая альтернатива), RTL не тестирует внутреннее состояние и props, а проверяет, что пользователь видит и может делать.

```jsx
import { render, screen, fireEvent } from "@testing-library/react";
import Counter from "./Counter";

test("увеличивает счётчик при клике", () => {
  render(<Counter />);

  const button = screen.getByRole("button", { name: /увеличить/i });
  expect(screen.getByText("Счёт: 0")).toBeInTheDocument();

  fireEvent.click(button);
  expect(screen.getByText("Счёт: 1")).toBeInTheDocument();
});
```

### 104. Как тестировать компонент React?

Для тестирования React-компонента: 1) рендерим компонент с помощью `render()` из RTL; 2) находим нужные элементы через `screen.getBy...` / `screen.findBy...` (по тексту, роли, test-id); 3) имитируем действия пользователя через `fireEvent` или `userEvent` (клик, ввод текста); 4) проверяем результат через `expect`. При необходимости мокаем API-вызовы, пропсы и контексты.

```jsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

function Greeting({ name }) {
  const [greeted, setGreeted] = useState(false);
  return (
    <div>
      <p>{greeted ? `Привет, ${name}!` : "Нажми кнопку"}</p>
      <button onClick={() => setGreeted(true)}>Поздороваться</button>
    </div>
  );
}

test("показывает приветствие после клика", async () => {
  render(<Greeting name="Андрей" />);

  expect(screen.getByText("Нажми кнопку")).toBeInTheDocument();

  await userEvent.click(screen.getByRole("button", { name: /поздороваться/i }));

  expect(screen.getByText("Привет, Андрей!")).toBeInTheDocument();
});
```

## Общие вопросы веб-разработки

### 105. Что происходит, когда пользователь вводит URL в браузере?

1. **DNS-запрос** — браузер преобразует доменное имя (example.com) в IP-адрес через DNS-сервер.
2. **TCP-соединение** — устанавливается соединение с сервером (TCP handshake), при HTTPS — ещё и TLS handshake.
3. **HTTP-запрос** — браузер отправляет GET-запрос на сервер.
4. **Ответ сервера** — сервер возвращает HTML-документ (и заголовки).
5. **Парсинг HTML** — браузер строит DOM-дерево, обнаруживает CSS и JS-ресурсы.
6. **Загрузка ресурсов** — CSS, JS, изображения загружаются параллельно.
7. **CSSOM + DOM → Render Tree** — строится дерево рендеринга.
8. **Layout (Reflow)** — вычисляются размеры и позиции элементов.
9. **Paint** — элементы отрисовываются на экране.
10. **JavaScript** — выполняется JS, может модифицировать DOM.

### 106. Что такое HTTP / HTTPS? Какие методы HTTP вы знаете?

**HTTP** (HyperText Transfer Protocol) — протокол передачи данных между клиентом (браузером) и сервером. **HTTPS** — его защищённая версия с шифрованием через TLS/SSL, которая обеспечивает конфиденциальность и целостность данных. Основные методы HTTP:
- **GET** — получить данные (не изменяет данные на сервере).
- **POST** — создать новый ресурс / отправить данные.
- **PUT** — полностью обновить ресурс.
- **PATCH** — частично обновить ресурс.
- **DELETE** — удалить ресурс.
- **HEAD** — как GET, но без тела ответа (только заголовки).
- **OPTIONS** — узнать, какие методы поддерживаются.

### 107. Что такое REST API?

REST (Representational State Transfer) — это архитектурный стиль для проектирования API. REST API использует HTTP-методы для операций над **ресурсами**, представленными URL-адресами. Основные принципы: 1) **клиент-серверная архитектура**; 2) **stateless** — каждый запрос содержит всю необходимую информацию; 3) **единообразный интерфейс** — ресурсы идентифицируются URL, операции — HTTP-методами; 4) ответы обычно в формате JSON.

```
GET    /api/users       — получить список пользователей
GET    /api/users/42    — получить пользователя с id=42
POST   /api/users       — создать нового пользователя
PUT    /api/users/42    — обновить пользователя 42
DELETE /api/users/42    — удалить пользователя 42
```

### 108. Что такое CORS?

CORS (Cross-Origin Resource Sharing) — это механизм безопасности браузера, который **ограничивает запросы** с одного домена (origin) к ресурсам на другом домене. Например, если фронтенд на `localhost:3000` делает запрос к API на `localhost:8080`, браузер заблокирует его, если сервер не разрешит это через CORS-заголовки. Сервер должен ответить заголовком `Access-Control-Allow-Origin`, указывающим, какие домены могут делать запросы. Это защита пользователя, а не сервера — серверные запросы CORS не подчиняются.

```
// Заголовки ответа сервера, разрешающие CORS
Access-Control-Allow-Origin: https://my-frontend.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
```

### 109. Что такое SPA (Single Page Application)?

SPA (одностраничное приложение) — это веб-приложение, которое загружает **одну HTML-страницу** и далее динамически обновляет содержимое без перезагрузки, используя JavaScript. При навигации контент меняется на стороне клиента: браузер подменяет часть DOM и обновляет URL через History API, но новая страница с сервера **не загружается**. Плюсы: быстрая навигация, плавный UX, офлайн-возможности. Минусы: первоначальная загрузка дольше, сложнее SEO. React, Vue, Angular — типичные инструменты для SPA.

### 110. Что такое SSR и CSR? В чём разница?

**CSR (Client-Side Rendering)** — HTML-страница приходит с сервера почти пустой, а весь контент рендерится **на стороне клиента** с помощью JavaScript. Пока JS не загрузится и не выполнится, пользователь видит белый экран. **SSR (Server-Side Rendering)** — HTML генерируется **на сервере** для каждого запроса и отправляется клиенту уже с контентом. Пользователь быстрее видит страницу, SEO лучше. Минус SSR — нагрузка на сервер. Фреймворки для SSR: **Next.js** (React), **Nuxt** (Vue). Также существует **SSG** (Static Site Generation) — HTML генерируется на этапе сборки.

### 111. Что такое localStorage, sessionStorage и cookies? В чём разница?

Все три используются для хранения данных в браузере:
- **`localStorage`** — хранит данные **без срока годности** (пока не удалишь явно). Объём ~5–10 МБ. Доступен только на клиенте.
- **`sessionStorage`** — хранит данные **на время сессии** (до закрытия вкладки). Объём ~5 МБ. Каждая вкладка имеет свой sessionStorage.
- **`cookies`** — хранят малые объёмы данных (~4 КБ). Автоматически **отправляются на сервер** с каждым HTTP-запросом. Можно задать срок годности, домен, путь, флаги (`HttpOnly`, `Secure`, `SameSite`).

```js
// localStorage
localStorage.setItem("theme", "dark");
const theme = localStorage.getItem("theme"); // "dark"
localStorage.removeItem("theme");

// sessionStorage — аналогичный API
sessionStorage.setItem("token", "abc123");

// cookies
document.cookie = "name=Андрей; max-age=3600; path=/";
```

### 112. Что такое JSON?

JSON (JavaScript Object Notation) — это текстовый формат обмена данными, основанный на синтаксисе объектов JavaScript. Он легко читается людьми и легко парсится машинами. JSON поддерживает: строки, числа, булевы, `null`, массивы и объекты. JSON — стандартный формат для обмена данными между клиентом и сервером в REST API. В JavaScript для работы с JSON есть два метода: `JSON.stringify()` (объект → строка) и `JSON.parse()` (строка → объект).

```js
// JavaScript-объект
const user = { name: "Андрей", age: 25, hobbies: ["React", "TypeScript"] };

// Преобразование в JSON-строку
const json = JSON.stringify(user);
// '{"name":"Андрей","age":25,"hobbies":["React","TypeScript"]}'

// Обратно в объект
const parsed = JSON.parse(json);
console.log(parsed.name); // "Андрей"
```

### 113. Что такое WebSocket?

WebSocket — это протокол, обеспечивающий **двустороннюю постоянную связь** между клиентом и сервером через одно TCP-соединение. В отличие от HTTP (запрос-ответ), WebSocket позволяет серверу **отправлять данные клиенту в любой момент** без запроса от клиента. Используется для: чатов, онлайн-игр, биржевых котировок в реальном времени, уведомлений, совместного редактирования. Соединение начинается как HTTP-запрос (handshake), а затем «обновляется» до WebSocket.

```js
const socket = new WebSocket("wss://example.com/chat");

socket.onopen = () => {
  console.log("Соединение установлено");
  socket.send(JSON.stringify({ text: "Привет!" }));
};

socket.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log("Получено:", data);
};

socket.onclose = () => {
  console.log("Соединение закрыто");
};
```

### 114. Что такое Progressive Web App (PWA)?

PWA (прогрессивное веб-приложение) — это веб-приложение, которое сочетает возможности веб-сайта и нативного мобильного приложения. Ключевые характеристики: 1) **установка** на домашний экран без магазина приложений; 2) **офлайн-работа** через Service Worker (кэширование ресурсов); 3) **push-уведомления**; 4) **быстрая загрузка** и плавный UX. Для PWA необходимы: HTTPS, файл **manifest.json** (описание приложения: иконки, цвет, имя) и **Service Worker** (скрипт, работающий в фоне для кэширования и перехвата запросов).

```json
// manifest.json
{
  "name": "Моё приложение",
  "short_name": "App",
  "start_url": "/",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#3498db",
  "icons": [
    { "src": "/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/icon-512.png", "sizes": "512x512", "type": "image/png" }
  ]
}
```
