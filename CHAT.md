## Ответы

### 1) `useReducer` vs кастомные хуки

Ты почти прав, но это не «или-или» — это вещи разного уровня. Они не конкурируют, а дополняют друг друга.

**`useReducer`** — это один конкретный хук для управления состоянием. Ты верно уловил его суть: он нужен, когда несколько переменных состояния **связаны между собой** и ты хочешь менять их **атомарно через один dispatch**, а не дёргать три `setState` по отдельности и рисковать промежуточным состоянием.

**Кастомный хук** — это совсем другое. Это не альтернатива `useReducer`, а **контейнер для любой переиспользуемой логики**. Внутри кастомного хука может быть что угодно:
- `useState` — если состояние простое
- `useReducer` — если состояние сложное
- `useEffect` — если нужны побочные эффекты
- другие кастомные хуки — если нужна композиция

Конкретный пример из твоего проекта: `useTasks` — это кастомный хук, внутри которого `useAsync` (ещё один кастомный хук), `useState`, `useEffect`, `useCallback`. Если бы логика фильтрации была сложнее, внутри мог бы быть и `useReducer`.

### 2) `.then/.catch` vs `async/await` в `useEffect`

Оба варианта рабочие, но есть нюанс.

**Проблема с `async/await`:** колбэк `useEffect` **не может быть `async` функцией напрямую**. Вот так нельзя:

```typescript
// ❌ НЕЛЬЗЯ — useEffect ожидает void или функцию cleanup, а не Promise
useEffect(async () => {
    const data = await fetchTasks();
    setTasks(data);
}, []);
```

React ожидает, что колбэк `useEffect` вернёт либо `undefined`, либо функцию очистки (cleanup). А `async` функция всегда возвращает `Promise` — React не знает что с ним делать и выдаст предупреждение.

**Если хочешь `async/await`**, нужно обернуть в отдельную функцию внутри:

```typescript
// ✅ Так можно
useEffect(() => {
    const load = async () => {
        try {
            const data = await fetchTasks();
            setTasks(data);
        } catch (e) {
            setError(e instanceof AppError ? e.message : ERROR_MESSAGES.NETWORK_ERROR);
        } finally {
            setIsLoading(false);
        }
    };
    load();
}, []);
```

**`.then/.catch`** работает без этой обёртки, потому что колбэк остаётся синхронной функцией, которая просто запускает цепочку промисов и возвращает `undefined`:

```typescript
// ✅ Тоже нормально
useEffect(() => {
    fetchTasks()
        .then(setTasks)
        .catch(e => setError(...))
        .finally(() => setIsLoading(false));
}, []);
```

**Что выбрать?** Для простых случаев (как у тебя — один fetch без условий) — `.then/.catch` компактнее и читается отлично. Для более сложной логики, где нужны последовательные запросы или условные ветки, `async/await` в обёртке удобнее. В индустрии используют оба варианта, строгого правила нет.

### 3) Зависимости `useEffect` — разбираем по частям

#### Почему `setIsLoading` не менялась?

Отличный вопрос! `setIsLoading` — это функция-сеттер, которую возвращает `useState`. React **гарантирует**, что эта функция **стабильна** — она создаётся один раз при монтировании компонента и **никогда не пересоздаётся**. Это написано в документации React:

> React guarantees that setState function identity is stable and won't change on re-renders.

Поэтому React разрешает не включать `setIsLoading` (и любой другой сеттер из `useState`) в массив зависимостей — линтер знает, что она не изменится.

В твоём `UserPage`:
```typescript
useEffect(() => {
    fetchUsers()
        .then(setUsers)             // setUsers — стабильна ✅
        .catch(e => setError(...))   // setError — стабильна ✅
        .finally(() => setIsLoading(false)); // setIsLoading — стабильна ✅
}, []);
```

Все три функции-сеттера (`setUsers`, `setError`, `setIsLoading`) стабильны, `fetchUsers` — это обычная функция, импортированная из модуля (она тоже не пересоздаётся, это просто ссылка на функцию из другого файла). Поэтому `[]` тут полностью корректен, и линтер не ругается.

#### Почему переданная функция `fn` может быть разной?

А вот теперь представь `useAsync`. Ты вызываешь его так:

```typescript
// Вариант 1: передаём ссылку на импортированную функцию
const { data } = useAsync(fetchUsers);
```

Здесь `fetchUsers` — это ссылка на функцию из модуля `fetchUsers.ts`. Она **не пересоздаётся** при каждом рендере (как и `setIsLoading`), потому что это один и тот же объект в памяти. Тут проблемы нет.

Но что если кто-то напишет так:

```typescript
// Вариант 2: передаём стрелочную функцию
const { data } = useAsync(() => fetchTasksByUserId(userId));
```

Вот тут проблема. Стрелочная функция `() => fetchTasksByUserId(userId)` **пересоздаётся при каждом рендере компонента**. Каждый рендер — новый объект функции в памяти.

Если `useAsync` внутри имеет `useEffect(() => { fn(); }, [fn])`, то при каждом рендере `fn` будет новой → `useEffect` сработает снова → новый запрос → новый рендер → бесконечный цикл.

#### В чём разница между `useRef` и `useCallback`?

Это совершенно разные инструменты, которые решают разные задачи. Но в контексте `useAsync` оба могут помочь со стабилизацией — отсюда путаница.

**`useRef`** — это «коробка» с полем `.current`, которая:
- живёт между рендерами (не уничтожается),
- **не вызывает перерисовку** при изменении `.current`,
- можно класть туда что угодно: DOM-элемент, число, функцию.

```typescript
const fnRef = useRef(fn);
fnRef.current = fn; // обновляем на каждом рендере — тихо, без ререндера

useEffect(() => {
    fnRef.current(); // всегда вызовет актуальную версию fn
}, []); // зависимость пустая — эффект сработает один раз
```

**`useCallback`** — это мемоизация функции. Он **возвращает ту же ссылку** на функцию, пока не изменились зависимости:

```typescript
const stableFn = useCallback(() => fetchTasksByUserId(userId), [userId]);
// stableFn не пересоздаётся, пока userId не изменится
```

Аналогия: `useCallback` — это как кеширование. «Не создавай новый объект, пока входные данные не изменились».

**Когда что использовать:**

| | `useRef` | `useCallback` |
|---|---|---|
| Что делает | Хранит мутабельное значение между рендерами | Мемоизирует функцию по зависимостям |
| Вызывает ререндер? | Нет | Нет (но возвращает новую ссылку, если зависимости изменились) |
| Типичное применение | Ссылка на DOM, хранение «предыдущего значения», свежая ссылка на колбэк внутри эффекта | Передача колбэка в дочерний компонент без лишних ререндеров |
| В контексте `useAsync` | Подход B: «спрятать» `fn` в ref, чтобы эффект не зависел от неё | Подход A: вызывающий код оборачивает `fn` в `useCallback`, чтобы ссылка была стабильной |

В твоём случае с `useAsync(fetchUsers)`, где `fetchUsers` — это импорт из модуля, **ни `useRef`, ни `useCallback` не нужны** — ссылка и так стабильна. Эти подходы нужны на случай, когда кто-то передаст стрелочную функцию (вариант 2 выше).

#### `useRef` vs `useCallback` — подробнее с примерами

Главное различие в одном предложении:
- **`useRef`** — мутабельный контейнер, невидимый для React (не участвует в рендеринге).
- **`useCallback`** — кешированная функция, видимая для React (участвует в сравнении зависимостей).

---

**Пример 1: `useRef` — сохранить предыдущее значение**

```typescript
function UserPage({ userId }: { userId: number }) {
    const prevUserId = useRef<number | null>(null);

    useEffect(() => {
        if (prevUserId.current !== null) {
            console.log(`Сменили пользователя: ${prevUserId.current} → ${userId}`);
        }
        prevUserId.current = userId;
    }, [userId]);

    return <div>User #{userId}</div>;
}
```

`useCallback` здесь **не подходит** — нам нужно хранить данные, а не функцию.

---

**Пример 2: `useRef` — доступ к DOM-элементу**

```typescript
function SearchInput() {
    const inputRef = useRef<HTMLInputElement>(null);

    const focusInput = () => inputRef.current?.focus();

    return (
        <>
            <input ref={inputRef} placeholder="Поиск..." />
            <button onClick={focusInput}>Фокус</button>
        </>
    );
}
```

`useCallback` здесь **бессмыслен** — задача не в стабилизации функции, а в получении ссылки на DOM-узел.

---

**Пример 3: `useCallback` — стабильный колбэк для дочернего компонента**

```typescript
function TasksPage() {
    const [tasks, setTasks] = useState<UserTask[]>([]);

    const toggleTask = useCallback((id: number, completed: boolean) => {
        setTasks(prev => prev.map(t =>
            t.id === id ? { ...t, completed: !completed } : t
        ));
    }, []);

    return (
        <div>
            {tasks.map(task => (
                <TaskCard key={task.id} task={task} onToggle={toggleTask} />
            ))}
        </div>
    );
}
```

Без `useCallback` → `toggleTask` каждый рендер новая → каждый `TaskCard` получает новый проп `onToggle` → все карточки перерендериваются.
С `useCallback` → `toggleTask` один и тот же объект → `TaskCard` (если обёрнут в `React.memo`) пропускает лишний рендер.

`useRef` здесь **не подходит** — нам нужна стабильная ссылка, которую React видит при сравнении пропсов. `ref.current` невидим для React: он не знает, менялась функция или нет.

---

**Пример 4: `useRef` — свежая ссылка внутри эффекта (паттерн из `useAsync`)**

```typescript
function useAsync<T>(fn: () => Promise<T>) {
    const [data, setData] = useState<T | null>(null);
    const fnRef = useRef(fn);
    fnRef.current = fn;

    useEffect(() => {
        fnRef.current()
            .then(setData);
    }, []);

    return data;
}

// Вызов — не нужен useCallback:
const data = useAsync(() => fetchTasksByUserId(userId));
```

Почему `useRef`, а не `useCallback`? Потому что задача — **спрятать** `fn` от `useEffect`, чтобы эффект не перезапускался. `useRef` невидим для массива зависимостей, а `.current` всегда актуален. Это паттерн «latest ref».

---

**Пример 5: `useCallback` — стабилизация `fn` снаружи (альтернатива примеру 4)**

```typescript
function useAsync<T>(fn: () => Promise<T>) {
    const [data, setData] = useState<T | null>(null);

    useEffect(() => {
        fn().then(setData);
    }, [fn]); // зависит от fn — эффект перезапустится, если fn изменится

    return data;
}

// Вызов — обязательно useCallback, иначе бесконечный цикл:
const loadTasks = useCallback(() => fetchTasksByUserId(userId), [userId]);
const data = useAsync(loadTasks);
```

Разница: здесь ответственность за стабильность лежит на **вызывающем коде**. Забыл `useCallback` → бесконечный цикл. Пример 4 (с `useRef`) безопаснее для потребителя.

---

**Итого — шпаргалка «когда что»:**

| Задача | Инструмент |
|---|---|
| Сохранить значение между рендерами без ререндера | `useRef` |
| Получить ссылку на DOM-элемент | `useRef` |
| Спрятать функцию от зависимостей `useEffect` (latest ref) | `useRef` |
| Передать стабильный колбэк в дочерний компонент | `useCallback` |
| Стабилизировать функцию для массива зависимостей | `useCallback` |
| Мемоизировать вычисление (не функцию, а значение) | `useMemo` (не `useRef` и не `useCallback`) |

### Зачем в `useTasks` отдельный `useState` + `useEffect`, а не просто `data` из `useAsync`?

Короткий ответ: потому что `toggleTask` **мутирует список задач локально**, а `data` из `useAsync` — read-only снимок с сервера.

Разберём подробно. Вот что происходит, когда пользователь нажимает чекбокс на задаче:

```typescript
const toggleTask = useCallback((id: number, completed: boolean) => {
    // ...
    updateTaskStatus(id, !completed).then(() => {
        setTasks(prev => prev.map(task =>
            task.id === id ? { ...task, completed: !completed } : task
        ));
    });
}, []);
```

`setTasks` меняет **локальный** массив задач — переворачивает `completed` у одной задачи. Это «оптимистичное обновление» — мы сразу показываем результат в UI, не дожидаясь полной перезагрузки с сервера.

Теперь представь, что мы убрали `tasks`/`setTasks` и используем `data` напрямую. У `useAsync` есть `setData`, но:

1. **`useAsync` — это универсальный хук.** Он не знает ни про задачи, ни про toggle. Если мы начнём возвращать из него `setData`, каждый потребитель сможет модифицировать загруженные данные как угодно — и хук перестанет быть «чистым загрузчиком». Это нарушение принципа единственной ответственности (SRP): `useAsync` отвечает за загрузку, а не за бизнес-логику мутаций.

2. **Два источника правды.** Если `useAsync` владеет данными и `useTasks` тоже может их менять через `setData`, непонятно, кто «хозяин» состояния. Сейчас всё чисто: `useAsync` загрузил → передал в `data` → `useTasks` скопировал в свой `tasks` и дальше управляет сам.

3. **Практический сценарий.** Допустим, позже мы добавим кнопку «Обновить список» (перезагрузить с сервера). `useAsync` вернёт новый `data`, `useEffect` сработает и перезапишет `tasks` свежими данными. Локальные мутации (toggle) и серверные данные — разделены.

Аналогия из Java: `useAsync` — это `Repository.findAll()`, возвращает snapshot. `tasks` в `useTasks` — это рабочая коллекция в сервисе, которую можно модифицировать. Ты не мутируешь результат `findAll()` напрямую — ты копируешь его в свою структуру и работаешь с ней.

**Если бы `toggleTask` не существовал** (задачи только читаются, никогда не меняются локально), то отдельный `useState` + `useEffect` был бы не нужен — можно было бы использовать `data` напрямую, как в `UserPage`:

```typescript
const { data: users, isLoading, error } = useAsync(fetchUsers);
```

Там нет мутаций — поэтому и нет копии.
